Implementation Plan: Product Polish and Delight — Learner-Facing Landing Hierarchy, Motion Feedback, Milestone Celebration, and Finishing Touches (spec/018-product-polish-and-delight.txt)

Plan status: draft pending tech-lead review

## Brief Summary

`spec/018` is a product-experience pass layered on top of the now-complete 014-017 work (tokens, altitude classes, action-color unification, the `component-kit` primitives, dark mode, icons). It touches no visual token and adds no component-kit primitive beyond what P2's celebration moment needs; instead it fixes four concrete gaps in the assembled product: (1) `CollectionLanding.tsx` renders a developer-facing template-validation PASS/FAIL badge on every learner-facing lecture card, (2) that same card list has no visual signal for "which lecture do I resume," (3) the product has almost no motion feedback anywhere and zero acknowledgment of finishing a lecture or a course, and (4) the site has no favicon/manifest/OG metadata and no styled error pages.

The plan is seven steps following the spec's P0 → P3 order, but two are combined because researching the actual code showed they share one root cause and one pure function: Step 1 removes the validation badge *and* replaces it with real learner progress state in the same change (spec P0 + the state-computation half of P1), because the badge's replacement is exactly the data the resume-highlight needs. Step 2 adds the resume-highlight and card-level hover motion (rest of P1). Steps 3-4 build the small motion-token vocabulary and the two completion acknowledgments (P2) — which, per the Corrections below, turn out to already have almost all their plumbing in place. Steps 5-6 are the P3 finishing touches (favicon/manifest/OG, then error pages). Step 7 is verification.

## Corrections To Spec 018's Assumptions

Several things the spec describes more simply than the code actually is, discovered by reading the real components rather than assuming from the spec's own Problem section. None change the plan's direction, but all change *how* a step should be implemented:

- **The "0 of 4 milestones reached" copy the spec's Problem section attributes to `LearningPathNavigation` actually lives in `progress/LectureProgressBar.tsx:39`** (`{milestones.filter(...).length} of {milestones.length} milestones reached`, with `milestones = [25, 50, 75, 100]` at line 6). `LearningPathNavigation.tsx` has no milestone concept at all — it only tracks the currently-visible section anchor. This doesn't change any goal, but it means Step 4's lecture-completion trigger point (`percentComplete` crossing 100) is already a first-class, named concept in `LectureProgressBar`, not something to derive fresh.
- **A milestone toast mechanism already exists and already fires on every section toggle — it is not P2 "greenfield" work, it's a P2 special-case.** `ProgressProvider.tsx`'s `toggleSection` (lines 194-212) calls `announce(message)` (lines 189-192) on *every* toggle, which sets both `announcement` and `toast` state; `ProgressLiveRegion.tsx` renders `toast` in a `.progress-toast` element (`globals.css:851-864`) animated by the exact `progress-toast-in` keyframe (`globals.css:880-889`) spec/018's Product Decisions point at. Today the message is the same generic "Section marked complete. X of Y sections finished." whether X/Y is 4/10 or 10/10 — confirming the spec's Problem statement exactly, but also meaning Step 4 is "special-case one `announce()` call when the new percentage is 100 and the old one wasn't," not "build a toast system."
- **The course-completion trigger cannot live in `CollectionProgressProvider`, because it never mounts at the moment a course actually gets completed.** Spec G4/P2 says the course-level acknowledgment is "triggered when `CollectionProgressProvider` reports 100%" — but `CollectionProgressProvider` only wraps `CollectionLanding` (the landing page), while the action that pushes a course to 100% (toggling the last section of the last incomplete lecture) happens on a *lecture* page, where only `ProgressProvider` is mounted. `ProgressProvider` already has everything needed to detect this in place, though: its existing collection-write effect (`ProgressProvider.tsx:154-187`) already builds the full up-to-date `collection: Record<string, LectureProgress>` map (line 166-176) before writing it to `localStorage`, and `calculateCollectionProgress(progress: CollectionProgress, lectures: ProgressLecture[])` (`src/lib/lecture-template/progress.ts:100`) is already a pure, imported-elsewhere function. Step 4 computes the collection summary before and after this effect's write (using the pre-existing `existing`/`collection` values already read from `localStorage` at that point) entirely inside `ProgressProvider`, and fires the course-complete acknowledgment from there — `CollectionProgressProvider` is not touched.
- **P1's "resume here" card state does not need new logic — it needs the existing `calculateResumeTarget` function called from one more place.** `CollectionPrimaryAction.tsx:5,11` already imports `calculateResumeTarget` from `@/lib/lecture-template/resumeTarget` and uses it to pick the hero CTA's label/href. `CollectionLanding.tsx` computes `progressLectures` already (lines 29-34) and could call the same function to find the one card that should be visually distinguished — but `ResumeTarget` (`resumeTarget.ts:3-6`) only exposes `{ label, href }`, not a `slug`, so matching it against `result.slug` in the `.map()` means either parsing the slug out of `href` (fragile: `href` is `/lectures/${slug}` or `/lectures/${slug}#${anchor}`) or adding a `slug: string` field to `ResumeTarget` and its return sites (`resumeTarget.ts:16,22,27,32`) — a small, additive, low-risk change, done in Step 2 rather than string-parsing in the component.
- **`calculateResumeTarget`'s three label states (`"Start course"`, `"Resume course"`, `"Continue course"`) map to two different card behaviors, not one.** `"Start course"` (nothing begun) and `"Resume course"` (partway through) both point at a real "go here next" card that should get the resume-highlight. `"Continue course"` (`resumeTarget.ts:32`) fires when every lecture is already 100% complete and falls back to `lectures[0]`'s href — highlighting lecture 1 as "next up" in that state would be wrong (the course is finished; Step 4's course-complete acknowledgment is the correct signal at that point, not a highlighted card). Step 2 must only apply the resume-highlight when `target.label !== "Continue course"`.
- **Removing the validation badge is lower-risk than the spec's cautious "update tests deliberately" framing suggests.** Grepping every file in `tests/lecture-template/` for `"validation-badge"`, `"PASS"`, and `"FAIL"` returns zero matches — no test asserts on the badge's class name or text today. Step 1 needs no test updates for the removal itself (only for whatever replaces it, if that gets a new testable class/text — see Step 1).
- **`.lecture-list-item` was never migrated to the `Card` component-kit primitive in spec 017.** It's a plain `<li className="lecture-list-item"><article>...` (`CollectionLanding.tsx:89-125`), not a `lecture-component` type, so it was correctly out of scope for 017's `Card` migration (which covered `lecture-kit` content components, not the collection-landing page shell). This matters for Step 2: the resume-highlight cannot be added via a `Card` `altitude` prop — it must be a plain CSS modifier class on `.lecture-list-item`, reusing the *pattern* `.surface-emphasis` already established (`globals.css:954-959`: `border-left-width: 7px` + `box-shadow: var(--shadow-sm)`) rather than the `Card` component itself.

## Proposed Implementation Steps

### Step 1: Remove the validation badge; replace it with learner progress state (spec P0 + G1)

- In `CollectionLanding.tsx:84-99`, delete the `statusClass`/`statusLabel`/`statusIcon` computation and the `validation-badge` `<span>`.
- Replace it with a small, text-labeled state derived from data `CollectionLectureProgress`/`CollectionProgressProvider` already computes for that slug: "Not started" / "In progress" / "Completed" (exact copy TBD in review, but must be visible text or an `aria-label`-carrying element, not color-only — see spec Accessibility Requirements). Since `CollectionLectureProgress` (`progress/CollectionProgressBar.tsx`) already renders a percentage-labeled mini progress bar per card (`CollectionLanding.tsx:123`), the simplest implementation is a short label next to/above that existing bar, computed from the same per-lecture progress percentage rather than a second, separate data read.
- Decide G7 explicitly: either (a) gate a `validation-badge`-equivalent behind an authoring-only path (e.g., a `?authoring=1` query flag checked in `CollectionLanding` that a learner would never navigate to, rendering the PASS/FAIL badge only in that mode), or (b) drop learner-visible validation status entirely, since `npm run validate`/`npm run doctor` already give the author a CLI-based validation view and may be sufficient. Recommend (b) for this pass — no evidence in the codebase today that an author previews `CollectionLanding` itself (rather than the CLI validator) to check validation status, and (a) adds a code path with no test coverage and no confirmed user. Record whichever is chosen; do not leave it implicit.
- No test updates required for the removal itself (confirmed zero matches on `validation-badge`/`PASS`/`FAIL` across `tests/lecture-template/`); add one small assertion (in `visual-hierarchy.test.tsx` or a new test) that the new learner-state label is present and the old badge text is not, so a future change can't silently reintroduce it.
- Run after this step: `npm run test -- tests/lecture-template/visual-hierarchy.test.tsx tests/lecture-template/collection-landing-anchor.test.tsx tests/lecture-template/collection-cta.test.tsx`.

### Step 2: Resume-highlight card state and card-level hover/focus feedback (spec P1)

- Add `slug: string` to `ResumeTarget` (`resumeTarget.ts:3-6`) and populate it at each of the three return sites (`resumeTarget.ts:16,22,27,32` — for the fallback/"Start course" and "Continue course" cases, `lectures[0].slug`; for "Resume course", the matched `lecture.slug`).
- In `CollectionLanding.tsx`, call `calculateResumeTarget(progress, progressLectures)` once (reading `progress` from the same `useCollectionProgress()`/`CollectionProgressProvider` context `CollectionLectureProgress` already reads inside this tree), yielding one `slug` to compare against `result.slug` in the existing `.map()` (line 80).
- Apply a resume-highlight modifier class to the one matching `.lecture-list-item` **only when `target.label !== "Continue course"`** (see Corrections) — implemented as a new CSS class reusing the `.surface-emphasis` pattern (`globals.css:954-959`: increased `border-left-width`, `box-shadow: var(--shadow-sm)`), applied to `.lecture-list-item` rather than routed through the `Card` component (which this element doesn't use — see Corrections).
- Apply a quieter "completed" modifier (muted border/checkmark, consistent with `LearningPathNavigation`'s existing `.nav-item-complete` treatment) to every card whose per-lecture progress is 100%.
- Add `:hover`/`:focus-within` to `.lecture-list-item` itself (`globals.css:2103-2109`) — a border-color and/or subtle `transform: translateY(-1px)`-style lift, using Step 3's new motion tokens (sequence Step 3 before finishing this rule, or land a temporary hardcoded value and update it once Step 3's tokens exist — either order works since it's the same file).
- Run after this step: `npm run test -- tests/lecture-template/collection-landing-anchor.test.tsx tests/lecture-template/collection-cta.test.tsx tests/lecture-template/collection-progress-integration.test.tsx`.

### Step 3: Motion vocabulary tokens

- Add 2-3 CSS custom properties near the existing token block at the top of `globals.css` (e.g. `--motion-fast: 160ms ease`, `--motion-standard: 180ms ease`, matching the values already in ad hoc use at lines 391, 652, 863, and in `reveal-fade-in`'s consumers), rather than a new arbitrary scale.
- Replace the four existing hardcoded duration/easing values (`globals.css:391`, `652`, `863`, and the `Disclosure`-consumed `reveal-fade-in` animation duration at `909`) with references to the new custom properties — a mechanical, behavior-preserving substitution, verifiable by confirming no computed-style diff.
- Apply the same tokens to Step 2's new `.lecture-list-item` hover/focus transition.
- Add every new animated/transitioned selector introduced in Steps 2 and 4 to the existing `prefers-reduced-motion: reduce` block's selector list (`globals.css:2418-2430`), following its established pattern of `transition: none; animation: none;`.
- Run after this step: `npm run test` (full suite is cheap here; this step only touches `globals.css` values, not markup, so a targeted subset isn't meaningfully faster) plus a manual visual diff at 390/768/1280px confirming no unintended change to the four pre-existing animated elements.

### Step 4: Lecture- and course-completion acknowledgment

- **Lecture-level**: in `ProgressProvider.tsx`'s `toggleSection` (lines 194-212), after computing `nextSummary`, special-case the message passed to `announce()` when `nextSummary.percentComplete === 100` and the previous `summary.percentComplete < 100` — e.g. "Lecture complete! All {totalSections} sections finished." instead of the generic per-toggle message. This reuses the existing `toast`/`ProgressLiveRegion`/`progress-toast-in` path unchanged; only the message and, optionally, an added modifier class on `.progress-toast` (e.g. `.progress-toast-milestone`) for a slightly more noticeable but still brief treatment differ.
- **Course-level**: in the same file's collection-write effect (lines 154-187), after building `collection` (the full up-to-date map, line 166-176) and before/alongside the existing `window.localStorage.setItem(collectionKey, ...)` call, compute `calculateCollectionProgress(collection, collectionLecturesRef.current)` and compare its `percentComplete` against the same calculation run on the *previous* stored value (read via the existing `existing`/`JSON.parse(existing)` at lines 165-175, before this toggle's write) to detect a 0→100 (well, `<100`→`100`) transition. On that transition, call a new `announceCourseComplete()` (parallel to the existing `announce()`, added to `ProgressContextValue`) so `LecturePage`'s already-mounted `ProgressLiveRegion` can render it — no new provider, no change to `CollectionProgressProvider` (see Corrections).
- Decide whether the course-complete message reuses the exact same `.progress-toast` element (simplest, recommended: it already supports arbitrary string content and the `160ms` entrance is appropriate) or needs a visually distinct treatment (e.g., a longer-lived or centered variant) — if the latter, this is the one place this spec might justify a second, small state in `ProgressLiveRegion.tsx` rather than a new component-kit primitive; keep it in `progress/` either way, since it's domain-specific (progress-toast display logic), not a generic primitive.
- Ensure both new toast variants are added to the `prefers-reduced-motion` selector list from Step 3 (they render through the same `.progress-toast` element, so likely already covered — confirm rather than assume).
- Persist "already shown" state using the mechanism the spec requires (fire once per milestone) — since `percentComplete` transitioning from `<100` to `100` is itself edge-triggered (only fires once per session unless progress is reset and re-completed, which is arguably correct — re-finishing after a reset deserves a second acknowledgment), no additional persisted "have I shown this" flag is needed beyond the transition check itself. Confirm this reasoning holds during implementation rather than assuming; if a learner can hit the 100% transition repeatedly without an intervening reset (e.g. toggling a section off and back on at the boundary), guard against a repeat toast within one render cycle.
- Run after this step: `npm run test -- tests/lecture-template/progress-provider.test.tsx tests/lecture-template/progress-model.test.ts tests/lecture-template/progress-ui-enhancement.test.tsx tests/lecture-template/collection-progress-integration.test.tsx`.

## Risks, Assumptions, And Open Questions

- **The exact visual/copy treatment of the two completion toasts (lecture vs. course) is a design judgment call this plan doesn't resolve** — Step 4 recommends reusing `.progress-toast` unchanged for both (simplest, lowest-risk), but flags that a visually distinct, more celebratory course-complete moment might be worth the small extra component surface. Decide by quick mockup/screenshot before implementing, not mid-Step-4.
- **G7's disposition (relocate vs. remove validation status from the learner view) is this plan's one open product decision**, resolved provisionally as "remove, rely on existing CLI validation tooling" in Step 1 — flag to whoever owns spec/018 if an author-facing in-browser validation view is actually wanted, since that would add a small amount of new scope (a gated route/flag) this plan currently treats as unnecessary.
- **The course-complete detection in `ProgressProvider.tsx` (Step 4) reads and re-parses the same `localStorage` collection entry the existing effect already reads** — this is a small amount of redundant work (two JSON parses instead of one) inside an already-debounced (`writeDelayMs`) effect, not a performance concern at this scale, but worth a one-line comment explaining why the before/after comparison needs the pre-write value.
- **No mechanical enforcement that new motion introduced in Steps 2 and 4 is added to the `prefers-reduced-motion` block** (`globals.css:2418-2430`) — same category of review-only enforcement noted as a risk in 017's plan for the `component-kit`/`lecture-kit` import boundary; flag in code review, don't assume it's automatic.
- **Step 2's resume-highlight assumes exactly one card should ever be emphasized at a time** — if a future change allows a learner to jump around non-linearly in a way `calculateResumeTarget`'s linear "first incomplete lecture in authored order" logic doesn't capture well, the highlight could point at a lecture the learner doesn't actually consider "next." Out of scope to fix here; `calculateResumeTarget`'s existing logic is reused as-is, not redesigned.
