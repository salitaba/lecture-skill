Plan status: draft pending tech-lead review

## Brief Summary

`spec/017` is a componentization pass layered on top of `spec/016-visual-design-system.txt` (already implemented in this checkout â€” tokens, altitudes, action-color unification, dark mode, and the `Icon.tsx` icon set all exist today). It introduces no new visual design, token, or authoring-schema decision; it extracts primitives that already exist *informally* â€” a card/surface wrapper, a button, the icon set, and a reveal/hide disclosure pattern â€” out of ~20-31 hand-rolled call sites in `src/components/lecture-kit/` and into a new `src/components/component-kit/` directory of domain-agnostic components that `lecture-kit` composes instead of re-deriving.

The plan is ten steps, following the spec's P0 â†’ P1 â†’ P2 â†’ P3 order: scaffold the directory, build and migrate `Button` (highest-evidence duplication, including one proven byte-identical CSS rule under two names), build and migrate `Card`, move `Icon.tsx` into the new directory, build and migrate `Disclosure`, then build and migrate three more primitives found on a second pass over the codebase â€” `ProgressMeter`, `RadioOptionGroup`, `LabeledSection` â€” before updating docs and verifying. Every step is additive to already-existing files (`globals.css`, ~28 `.tsx` files) with one deliberate exception called out in Step 2 (a literal class-name/test-string change, chosen rather than avoided).

## Corrections To Spec 017's Assumptions

Two things the spec states more simply than the code actually is; neither changes the plan's direction, but both need a decision made explicitly during implementation rather than assumed away:

- **There are five button shapes in `globals.css`, not three.** The spec's G2/Recommended Architecture describes `primary`, `ghost`, and an anchor-CTA mode. Reading the actual CSS turns up two more: `.progress-reset-button` (`globals.css:704-726`) is bordered like the `ghost` family but its hover state changes background to `--surface-accent` rather than filling with the accent color the way `.tabs-list button`/`.checklist-reset`/`.flashcard-reveal`'s shared hover rule does (`globals.css:1533-1538`) â€” a possibly-unintentional drift, not a designed distinction. `.resume-prompt-dismiss` (`globals.css:828-836`) is a third shape again: no border, no background, underlined text. Neither is a blocker, but Step 2 below makes an explicit, recorded call on each rather than silently forcing all three into one `ghost` variant.
- **Two buttons don't fit `primary`/`ghost` at all, and shouldn't be forced to.** `.theme-toggle` (`globals.css:175-201`, a 44px circular icon button with a shadow) and `.section-completion-toggle` (`globals.css:749-772`, a 34px circular toggle with a checkmark glyph) are both genuinely circular icon-only controls, structurally unlike either pill-shaped variant. The spec's own Product Decision says not to manufacture a variant to avoid a hard call â€” so this plan explicitly excludes `ThemeToggle.tsx` and `SectionCompletionToggle.tsx` from the `Button` migration in Steps 2, and says so up front here so a reviewer reads it as a decision, not a missed call site.
- **`Icon.tsx` has 4 import sites, not "~2."** Spec G4 estimates roughly two; the actual consumers are `GlossaryIndex.tsx:2`, `ThemeToggle.tsx:4`, `LearningPathNavigation.tsx:7`, and `LecturePage.tsx:3` (which imports via the absolute alias `@/components/lecture-kit/Icon` while the other three use the relative `./Icon` â€” Step 4 updates all four to point at the new location).
- **`Diagram.tsx` and `Quote.tsx` are the two real exceptions to the `Card` shape**, not variations that fit cleanly: both render `<figure>` instead of `<aside>`, neither has an `<h3>` title (`Quote` has no title at all; `Diagram` uses `<figcaption>` for its title, a different slot entirely), and `Diagram` has no altitude class at all today (`lecture-component diagram-card` â€” no `surface-*`). `Card`'s prop shape (Step 3) is designed around these two as the boundary cases, not as an afterthought.
- **Three more primitives surfaced on a second audit pass, after the first draft of `spec/017` stopped at `Button`/`Card`/`Icon`/`Disclosure`.** All three follow the exact same "CSS already merged, component layer never followed" evidence pattern the spec already uses for `Button`: `.lecture-progress-bar, .collection-progress-bar, .collection-progress-mini-bar` (`globals.css:637-639`) and `.lecture-progress-fill, .collection-progress-fill` (`globals.css:649-650`) are already one shared CSS rule across three hand-rolled React implementations (`ProgressMeter`, Step 6); `.assessment-region` and `.worked-example-region`/`.mistake-why` (`globals.css:1701-1717`) already share nearly identical padding/border-top rules across 15 hand-written `<h4>`-labeled sections in four files (`LabeledSection`, Step 8); and `.quiz-option`/`.assessment-option` are two separately-declared, nearly-identical rules with one likely-unintentional divergence (`align-items: center` vs. `flex-start`) that a shared `RadioOptionGroup` component (Step 7) would have prevented from ever drifting apart. These are added as spec P3 and plan Steps 6-8, inserted before the docs/verification steps that were originally Steps 6-7.

## Proposed Implementation Steps

### Step 1: Scaffold `src/components/component-kit/`

- Create `src/components/component-kit/` with an `index.ts` barrel that re-exports each primitive as it's added in later steps (`export * from "./Button"`, etc.).
- Establish the rule this plan enforces by review, not tooling (no eslint plugin change in this pass â€” see Risks): nothing under `component-kit/` imports from `lecture-kit/` or `@/lib/lecture-template`.
- No call-site changes yet; this step only creates the empty home.

### Step 2: Build `Button` and migrate its call sites (spec P0)

- New file `src/components/component-kit/Button.tsx`. Two variants, chosen on the evidence in "Corrections" above:
  - `primary` â€” filled `--action`, merges `.quiz-reveal-button` (`globals.css:1200-1216`), `.assessment-reveal-button` (`globals.css:1369-1385`, byte-identical to the previous rule today), `.resume-prompt-action` (`globals.css:811-826`), and `.collection-primary-action` (`globals.css:1961-1979`, larger padding/font-size â€” confirm during implementation whether that's an intentional size step for a bigger CTA or drift; if intentional, expose it as a `size` prop rather than a second variant).
  - `ghost` â€” bordered, accent-fill-on-hover, merges the already-partially-shared `.tabs-list button` / `.checklist-reset` / `.flashcard-reveal` rule (`globals.css:1515-1538`). Fold `.progress-reset-button` (`globals.css:704-726`) in only if its muted, non-accent-fill hover is judged unintentional drift after a side-by-side screenshot; otherwise keep it as a `ghost` sub-case via an additional modifier class rather than inventing a third variant. Fold `.resume-prompt-dismiss` (`globals.css:828-836`, text-underline) in the same way.
  - The component must spread `...rest` onto the rendered element so `Tabs.tsx`'s `role="tab"`, `aria-selected`, `tabIndex`, `id`, and `onKeyDown` continue to work unmodified â€” `Button` styles and forwards, it does not own tab semantics.
  - `as="a"` mode (or a thin `Button.Link`) covers the two CTA-link call sites (`ResumePrompt`'s "Continue reading", `CollectionPrimaryAction`).
  - Explicitly excluded, per Corrections: `ThemeToggle.tsx`, `progress/SectionCompletionToggle.tsx` â€” both stay bespoke.
- **Make the class-name decision before migrating `Quiz.tsx`**: either (a) `Button` accepts a `className` override and every call site passes its legacy class name through unchanged (`className="quiz-reveal-button"`, etc.) so no test or noscript string needs to change, or (b) pick one canonical class (e.g. `btn-primary`) and update the 3 files that reference the old name literally (`Quiz.tsx`'s noscript `<style>` string, `tests/lecture-template/lecture-components.test.tsx:275`, and confirm `collection-cta.test.tsx`/`visual-hierarchy.test.tsx` if `collection-primary-action` is renamed too). Recommend (a) for this pass â€” it gets the component-layer de-duplication (one `Button` component, one merged CSS rule per variant) without also taking on a rename, which the spec's own Product Decisions say to avoid unless "explicitly called for." A rename can be a fast follow-up once `Button` has proven itself.
- Migrate call sites lowest-risk first: `Checklist.tsx` reset button â†’ `progress/LectureProgressBar.tsx` reset button â†’ `Flashcard.tsx` reveal â†’ `Tabs.tsx` tab buttons (verify keyboard nav still works, since `onKeyDown`/`tabIndex` must pass through) â†’ `FreeResponse.tsx` â†’ `PracticeTask.tsx` (both buttons) â†’ `QuestionSet.tsx` (per-question button inside `.map()`) â†’ `Quiz.tsx` (last, because of the noscript coupling â€” verify the literal test string still matches) â†’ `progress/ResumePrompt.tsx` (both buttons) â†’ `CollectionPrimaryAction.tsx`.
- Merge the corresponding `globals.css` rules as each call site migrates, rather than all at once, so a broken merge is traceable to one component.
- Run after this step: `npm run test -- tests/lecture-template/quiz-interaction.test.tsx tests/lecture-template/lecture-components.test.tsx tests/lecture-template/assessment-hierarchy.test.tsx tests/lecture-template/collection-cta.test.tsx tests/lecture-template/visual-hierarchy.test.tsx tests/lecture-template/advanced-components-interaction.test.tsx`.

- Migrate the 18 clean-fit components first, simplest first: `ConceptCard.tsx`, `StepList.tsx`, `Summary.tsx`, `Timeline.tsx` (single-purpose, near-zero conditional logic) â†’ `Callout.tsx`, `Accordion.tsx`, `Checklist.tsx`, `Comparison.tsx`, `GlossaryTerm.tsx` (verify the forwarded `id` prop still reaches the wrapper), `InstructorNote.tsx`, `MistakeCorrection.tsx`, `ResourceLinks.tsx` â†’ `PracticeTask.tsx`, `QuestionSet.tsx`, `Quiz.tsx` (verify `id={component.anchor}` still applied), `Tabs.tsx`, `WorkedExample.tsx`, `FreeResponse.tsx`.

### Step 4: Move `Icon.tsx` into `component-kit/` (spec P1, G4)

### Step 5: Build `Disclosure` and migrate its call sites (spec P2)

- New file(s): a `useDisclosure()` hook (owns `useId()`-derived trigger/region ids, `useState` boolean, and the `aria-expanded`/`aria-controls` pair) plus a thin `Disclosure` component that renders a `Button variant="primary"` trigger and a controlled region, for call sites where the revealed content is a plain block.
- Straightforward migrations: `Flashcard.tsx` (1 disclosure), `FreeResponse.tsx` (1), `QuestionSet.tsx` (N â€” one `Disclosure` instance per question inside the existing `.map()`, ids derived from the existing per-question `baseId` pattern). `PracticeTask.tsx` needs two independent instances on one component (hints, solution) â€” confirm the hook can be called twice in one component without id collisions (it can; `useId()` is unique per call site).
- `Quiz.tsx` is the spec-flagged judgment call (spec P2: "leave wired through the same primitive if its feedback-state logic fits... otherwise document why it stays bespoke"). Recommendation: share the `useDisclosure()` *hook* (state + ids) but keep `Quiz.tsx`'s own JSX for the trigger and revealed region, since reveal also disables the radio inputs (`disabled={revealed}`) and gates the correct/incorrect feedback message â€” domain logic that doesn't belong in a generic `Disclosure` component. This gets the boilerplate reduction without distorting the primitive to fit one caller's extra behavior.
- Verify the 016 motion work (`prefers-reduced-motion`-gated transitions on `.quiz-answer`, `.assessment-hidden-region`, `.flashcard-answer`) is unaffected â€” `Disclosure` changes who owns the boolean and renders the trigger, not the CSS applied to the revealed region, so no `globals.css` change should be needed here.
- Run after this step: `npm run test -- tests/lecture-template/quiz-interaction.test.tsx tests/lecture-template/advanced-components-interaction.test.tsx tests/lecture-template/assessment-hierarchy.test.tsx tests/lecture-template/quiz-feedback-styles.test.tsx`.

## Data Model, API, Persistence, Event, Or Domain Changes

- Template schema, parser, validator, renderMarkdown, progress model: no changes.
- New internal component props (`Button`, `Card`, `Disclosure`, `ProgressMeter`, `RadioOptionGroup`, `LabeledSection`'s prop surfaces) are implementation details of `lecture-kit`'s rendering, not part of the authoring schema or any type exported to template authors.
- New files: `src/components/component-kit/{index.ts,Button.tsx,Card.tsx,Icon.tsx,Disclosure.tsx,ProgressMeter.tsx,RadioOptionGroup.tsx,LabeledSection.tsx}` (presentational/behavioral, no persistence).

## Risks, Assumptions, And Open Questions

- **No mechanical enforcement of the component-kit/lecture-kit import boundary.** `eslint.config.mjs` has no import-restriction plugin today, and adding one is outside this plan's scope (a devDependency addition, which the spec's Non-Goals don't explicitly forbid but which this plan treats as a separate decision). The boundary is enforced by code review only; flag to the team if they want it made mechanical in a follow-up.
- **The `primary`-variant class-name decision (Step 2) is the one place this plan takes on real risk.** Whichever of options (a) preserve-all-legacy-classnames or (b) rename-and-update-3-files is chosen, it must be decided and recorded before migrating `Quiz.tsx`, not discovered mid-migration â€” a partial migration that mixes both approaches would leave the codebase less consistent than before this spec started.
- **`.progress-reset-button` and `.resume-prompt-dismiss`'s exact fit into `ghost` is a visual judgment call**, not a foregone conclusion â€” Step 2 requires a side-by-side screenshot before deciding whether their hover/shape differences from the already-merged `ghost` trio are intentional (keep as sub-cases) or drift (fully unify). Do not force unification if a real, intentional visual distinction turns up.
- **`ThemeToggle`/`SectionCompletionToggle` exclusion from `Button`** is this plan's reading of the spec's own "don't manufacture a fourth variant" instruction, applied to two call sites the spec's Recommended Architecture section doesn't explicitly address. Flag this interpretation to whoever owns `spec/017` so the spec's prose can be corrected in a follow-up edit (the same pattern 016's plan used for its own spec-text correction) â€” it does not block implementation.
- **`Diagram.tsx`'s noscript CSS coupling to `.diagram-card`** is the same kind of literal-string risk as `Quiz.tsx`'s in Step 2, just lower-stakes since no test currently asserts on it (only the component's own noscript behavior would silently break if `Card` ever dropped or renamed that class) â€” verify manually with JavaScript disabled after migrating `Diagram` in Step 3.
- **No visual-regression tooling exists** (same constraint 016's plan documented) â€” all before/after comparisons in this plan are manual screenshots, not automated diffs.
- **`RadioOptionGroup`'s `align-items` divergence (Step 7) is a visual judgment call**, same category as the `Button` sub-shape calls in Step 2 â€” resolve by screenshot before merging the CSS, don't default to "just pick one" without checking whether `QuestionSet`'s longer option text actually needs top-alignment.
- **`ProgressMeter` (Step 6) is the lowest-risk step in this plan** â€” its CSS is already fully unified across all three call sites, so this step is close to a pure "extract the existing JSX into one component" refactor with minimal design judgment required, unlike `Button`/`RadioOptionGroup` where the CSS itself needs a decision first.
