Implementation Plan: Lecture Site Engine

Spec: spec/001-lecture-site-engine.txt
Plan status: finalized after tech-lead review

## Brief Summary

Build a new local-first lecture site engine from an empty repository. The MVP is a Next.js/npm app that reads one supported template file at `content/lecture.template.md`, validates the rigid lecture schema, renders a responsive lecture page at `http://localhost:3000`, and provides Codex-oriented instructions for converting raw lecture text into the supported schema.

The launch-critical promise is not only rendering a hand-authored template. The project must also support the documented golden conversion workflow where fresh Codex runs can convert `examples/raw-lecture.txt` into a validation-passing `content/lecture.template.md` without reading `examples/golden.template.md` during generation.

## Current Repository State

The repository currently contains only:

- `spec/001-lecture-site-engine.txt`: product and acceptance specification.

There is no existing package, app, source tree, tests, docs, or git metadata in the workspace. Implementation should scaffold the project from scratch while keeping generated dependencies out of source control unless the implementer later initializes normal project metadata.

## Technology Choices

- Use Next.js with the App Router, React, and TypeScript.
- Use npm scripts exactly as required by the spec:
  - `npm run dev`: local preview at `http://localhost:3000`.
  - `npm run validate`: validates `content/lecture.template.md` and exits nonzero on errors.
- Declare the runtime baseline in `README.md` as Node.js 24 LTS and bundled npm.
- Declare the runtime baseline in `package.json` with `engines.node` set to `>=24`.
- Prefer a position-preserving parsing stack:
  - Pre-scan the source text for frontmatter and fenced-block line ranges before parsing YAML.
  - Parse frontmatter and component YAML with `yaml.parseDocument` or an equivalent API that exposes line/column diagnostics.
  - Parse Markdown with `remark`/`unified` AST position data, or a carefully bounded tokenizer that explicitly tracks line numbers for headings, lists, fenced blocks, and section content.
  - Do not rely on frontmatter or Markdown helpers that discard source locators needed for author-locatable validation errors.
- Use a test runner that works cleanly with TypeScript and npm, such as Vitest. Keep parser/validator tests outside browser dependencies.

## Template Parsing And Validation Plan

The parser should produce a structured intermediate result, even when validation later fails:

- `frontmatter`: raw parsed frontmatter plus source locator metadata.
- `preOverviewContent`: body content before the first required heading.
- `overview`: heading locator and Markdown block content.
- `objectives`: heading locator and bullet items.
- `sections`: ordered section objects containing title, title locator, body blocks, component blocks, and generated anchor.
- `takeaways`: heading locator and bullet items.
- `errors`: syntax-level parse errors that prevent reliable validation of some content.

Validation should return:

- `valid`: boolean.
- `template`: normalized template only when all blocking validation passes.
- `errors`: list of author-locatable validation errors.

Each validation error should include:

- `code`: stable identifier for tests and future docs.
- `message`: concise problem statement.
- `locator`: line number when available, otherwise section/component/frontmatter context.
- `field`, `sectionTitle`, `componentType`, or `heading` when known.
- `hint`: short mechanical fix when applicable.

Implement P0 validation rules:

- Required frontmatter fields `title`, `description`, `audience`, `duration`, and `level` are present and non-empty after trimming.
- `level` is one of `beginner`, `intermediate`, or `advanced`.
- Unknown frontmatter fields are allowed and ignored.
- Required headings appear in order: `## Overview`, `## Learning Objectives`, one or more `## Section: <section title>`, `## Key Takeaways`.
- Duplicate `Overview`, `Learning Objectives`, or `Key Takeaways` headings fail validation.
- Non-empty content before `## Overview` fails validation.
- `Overview` contains at least one non-empty paragraph.
- `Learning Objectives` and `Key Takeaways` each contain at least one Markdown bullet item.
- Each section heading has a non-empty title.
- Each section contains at least one meaningful content block.
- `lecture-component` fenced blocks are allowed only inside sections.
- Unsupported component types fail validation and include unsupported type plus section context.
- `callout`, `concept_card`, `step_list`, and `code_block` payloads validate exactly against their required fields and allowed values.
- Empty `step_list.steps` fails validation.
- Invalid frontmatter YAML, malformed component YAML, and unclosed `lecture-component` fences produce blocking author-locatable errors.
- Unknown non-component fenced code blocks are valid normal Markdown content inside sections.
- Structurally parseable templates aggregate all detectable P0 errors in one run.

Do not silently coerce unsupported aliases, heading names, component types, or component payload shapes. The rigid schema is intentional for AI-agent reliability.

Add explicit locator tests for invalid frontmatter YAML, malformed component YAML, unclosed `lecture-component` fences, invalid component payloads inside named sections, non-empty content before `Overview`, and empty section titles. Tests should assert that errors include a line number when the parser can know one, or a precise section/component/frontmatter context otherwise.

## Rendering Plan

The root route should read and validate `content/lecture.template.md` on each local render. Implement `src/app/page.tsx` as a server component that uses Node-only file access through `src/lib/lecture-template/readTemplate.ts`, shares the same parser/validator path as `scripts/validate.ts`, and forces dynamic rendering, for example with `export const dynamic = "force-dynamic"` or an equivalent documented Next.js setting. If validation fails, it must render `ValidationScreen` instead of partial lecture content.

If validation passes:

- Render lecture metadata with `LectureHeader`.
- Render an ordered section navigation from the parsed section list.
- Generate stable unique anchors for duplicate section titles, such as `first-title`, `first-title-2`, and `first-title-3`.
- Render `Overview`, `Learning Objectives`, each ordered section, and `Key Takeaways`.
- Render supported `lecture-component` blocks only through components in `src/components/lecture-kit/`.
- Render normal Markdown paragraphs, lists, tables if present, and regular fenced code blocks without horizontal page overflow.
- Keep all base lecture UI component implementation under `src/components/lecture-kit/`.
- Render Markdown through React components without `dangerouslySetInnerHTML`. Do not enable raw HTML rendering such as `rehype-raw` unless the implementation also sanitizes it. Raw HTML in lecture Markdown should be escaped or rendered as inert text so `<script>` and inline HTML cannot inject executable or unexpected markup.

Responsive and accessibility behavior:

- Use semantic heading order and landmarks where practical.
- Ensure page width does not overflow at 390px or 1280px.
- Make section navigation keyboard usable with visible focus states.
- Keep code blocks horizontally scrollable within their own container rather than expanding the page.
- Use color contrast suitable for WCAG AA normal text.

## Documentation And AI Skill Plan

`README.md` must document:

- What the engine does.
- Node.js 24 LTS runtime expectation.
- `npm install`, `npm run dev`, `npm run validate`, and `http://localhost:3000`.
- Template paths and the recommended custom raw source path `content/raw-lecture.txt`.
- The exact schema, required frontmatter fields, required headings, supported component types, and fenced-block syntax.
- A complete synthetic example template that is not derived from `examples/raw-lecture.txt`.
- Common validation errors and how to fix them.
- The human workflow for asking an AI agent to follow `SKILL.md`.
- Links to `docs/mvp-review-checklist.md` and `docs/golden-conversion-batch.md`.
- How to preview `examples/component-demo.template.md` by copying it to `content/lecture.template.md`, including the warning that this replaces the active template and how to preserve or restore the active template.

`SKILL.md` must instruct Codex to:

- Read one raw lecture source.
- Extract title, audience, level, duration, objectives, sections, examples, caveats, practical steps, and takeaways from the source.
- Preserve source-grounded meaning and uncertainty.
- Avoid unsupported facts, statistics, tools, external references, or promises.
- Produce only the supported schema at `content/lecture.template.md`.
- Use only supported component types and exact payload shapes.
- Run `npm run validate` and iterate until validation passes.
- Optionally run or describe `npm run dev`.

The README and `SKILL.md` may mention golden file paths and protocol, but must not reproduce, summarize section-by-section, or closely paraphrase the content of `examples/golden.template.md` or the expected conversion of `examples/raw-lecture.txt`.

## Example And Review Artifacts Plan

Create:

- `examples/raw-lecture.txt`: one realistic plain-English lecture source, roughly 500-4,000 words, with at least one idea/process/caveat/code example/concept that naturally supports a visual component.
- `examples/golden.template.md`: valid template generated from the raw source for reviewer comparison after generation.
- `content/lecture.template.md`: valid active template for immediate preview. It can match the golden template in the shipping/demo state, but this must be called out as something to sanitize before golden evaluation.
- `examples/component-demo.template.md`: parse-valid template demonstrating `callout`, `concept_card`, `step_list`, and `code_block`.
- `examples/invalid/*.template.md`: fixtures for each P0 invalid case. Names should map directly to requirements, such as `missing-frontmatter-title.template.md`, `invalid-level.template.md`, `unsupported-component-type.template.md`, and `unclosed-lecture-component-fence.template.md`.
- `docs/mvp-review-checklist.md`: explicit `[ ] Pass / [ ] Fail` status fields covering metadata, objectives, section count, visual component usefulness, takeaways, and exact source-fidelity items confirming that every major claim is supported by `examples/raw-lecture.txt`, important source examples/terminology/caveats/practical steps are preserved unless omitted for clear concision, no unsupported facts/statistics/tools/external references/promises are added, uncertainty or incompleteness is preserved neutrally rather than resolved by invention, and reorganization does not change the source meaning.
- `docs/golden-conversion-batch.md`: form-like launch record with Codex product surface, model identifier or unavailable note, date, repository state, sanitized workspace setup, validation output per run, proof golden was inaccessible, starting active template status, reviewer checklist result, and pass/fail summary.
- `docs/educator-usefulness-log.md`: non-blocking quality-bar evidence template.

## Test Plan

Add focused automated tests for parser/validator behavior:

- Valid golden template passes.
- Valid component demo passes.
- Missing or empty required frontmatter fields fail with exact field names.
- Invalid `level` fails with allowed values.
- Missing required sections fail with section names.
- Out-of-order required headings fail.
- Duplicate singleton headings fail.
- Non-empty content before `Overview` fails.
- Empty `Overview`, `Learning Objectives`, and `Key Takeaways` fail with useful context.
- Empty section title fails.
- Section with no meaningful content block fails.
- `lecture-component` outside a section fails.
- Unsupported component type fails with type and section.
- Each supported component type validates required fields.
- Invalid `callout.variant` fails with allowed values.
- Empty `step_list.steps` fails.
- Invalid frontmatter YAML fails with locator.
- Malformed component YAML fails with component/section context.
- Unclosed `lecture-component` fence fails with locator.
- Duplicate section titles receive unique anchors that link to the correct section instances.
- Raw HTML or `<script>` in lecture Markdown is escaped or rendered inert and cannot inject executable markup.

Add CLI and integration-style tests where practical:

- `scripts/validate.ts` validates `content/lecture.template.md` by default.
- The validation CLI exits `0` and prints a success message for a valid template.
- The validation CLI exits nonzero for invalid input and prints field, section, component, locator, and remediation hints where applicable.
- Invalid active template causes the blocking validation screen model/component to receive all validation errors.
- Valid parsed template produces nav entries for every section in order.

Manual launch and quality checks:

- `npm install` on Node.js 24 LTS.
- `npm run validate`.
- `npm run dev` and open `http://localhost:3000`.
- Manual 390px and 1280px viewport checks for no horizontal overflow and usable section navigation.
- Apply `docs/mvp-review-checklist.md` to launch-counted generated golden lectures.
- Run the three-run golden conversion batch with sanitized access and record it in `docs/golden-conversion-batch.md`.
- For each golden conversion run, create a clean generation workspace, remove or hide `examples/golden.template.md`, replace or remove any golden-revealing `content/lecture.template.md`, verify README and `SKILL.md` do not contain section-by-section golden content, run the exact prompt from the spec, run `npm run validate`, restore the reviewer copy of `examples/golden.template.md` only after generation and validation, then record the setup and proof in `docs/golden-conversion-batch.md`.

## Implementation Order

1. Scaffold the Next.js/TypeScript/npm project with minimal route, styles, validation script, test runner, base directories, `package.json` scripts, and `engines.node >=24`.
2. Implement lecture template types, file reading, parsing, validation errors, and anchor generation.
3. Add P0 validation fixtures and automated tests, then wire `npm run validate`.
4. Implement the lecture-kit components, valid render path, and blocking validation screen.
5. Add valid content, golden/example/demo fixtures, and invalid fixtures.
6. Write README, `SKILL.md`, checklist, golden batch record, and educator usefulness log.
7. Run automated validation/tests and manually verify local preview and responsive behavior.
8. Run the golden-leakage preflight: confirm `README.md` and `SKILL.md` do not expose the golden answer, document how to remove or hide `examples/golden.template.md`, and document how to replace or remove golden-revealing `content/lecture.template.md` for each run.
9. Run and record the golden three-run launch-decision batch only after the implementation, docs, and sanitized workflow are ready.

## Risks And Mitigations

- Golden-answer leakage risk: README, `SKILL.md`, and synthetic examples must not reveal the expected conversion of `examples/raw-lecture.txt`. Keep the README example generic, keep golden-specific substance only in `examples/golden.template.md`, and make the sanitized workspace setup an explicit launch verification step.
- Parser brittleness risk: rely on Markdown/YAML AST tooling where possible and keep parser tests tied to invalid fixtures.
- Cascading syntax error risk: MVP allows one high-quality blocking error when malformed syntax prevents reliable later parsing; document and test the intended behavior.
- Markdown injection risk: lecture templates are local files, but raw HTML should still be escaped or sanitized to prevent surprising preview behavior.
- App Router caching risk: force dynamic rendering for the root route so local template edits appear during preview without a rebuild.
- Component duplication risk: keep all lecture-specific UI in `src/components/lecture-kit/` and make route files compose those components only.
- Launch protocol risk: the default active template may reveal the golden answer. The golden conversion record must explicitly document how `content/lecture.template.md` and `examples/golden.template.md` were made inaccessible or replaced before each generation run.
- Scope risk: do not implement browser-only authoring, LLM API integration, hosted deployment automation, diagrams, search, PDF export, slide mode, or deferred components in MVP.

## Assumptions And Open Questions

- Assumption: Because the repository is empty, introducing Next.js, TypeScript, and Vitest is acceptable and matches the spec's recommended technical direction.
- Assumption: Static export is not required for MVP.
- Assumption: The default shipping `content/lecture.template.md` may match the golden template for immediate preview, provided the golden conversion launch batch uses a sanitized workspace.
- Open question: Should the implementation include `npm run test` in addition to the required `npm run validate`? Recommendation: yes, because P0 validation coverage is launch-critical and should be executable independently.
- Open question: Should the MVP include Playwright for responsive checks? Recommendation: keep responsive checks manual for MVP unless implementation time allows adding a lightweight automated smoke test.
