Implementation Plan: Progress Tracking (spec/012-progress-tracking.txt)

## Summary

Add local-first learner progress tracking to the Lecture Site Engine. Learners can mark authored lecture sections complete, see a lecture progress bar and summary, reset progress, resume after returning to the page, and see per-lecture/course progress in collection mode. All state stays in browser `localStorage` under namespaced keys and must degrade to the normal static reading experience when JavaScript or storage is unavailable.

This is runtime learner convenience only. It does not change the authoring schema, template validation rules, review package source artifacts, accounts, sync, grading, analytics, gating, import/export, or server-side storage.

## Product Decisions For P0

- Progress tracks only authored `lecture.sections`, not Overview, Learning Objectives, Key Takeaways, answer keys, or assessment reveal state.
- Single lecture storage uses `lecture-progress:{lectureId}`. In this repository, use a stable sanitized id derived from `templatePath` plus lecture metadata title fallback so the same template keeps the same key across reloads.
- Collection storage uses `lecture-progress:collection:{collectionId}` and stores `{ [lectureSlug]: { [sectionAnchor]: boolean } }`. Use a stable collection id derived from collection base/course metadata context already available to validation, falling back to `"default"` when no better identifier exists.
- The provider will be a client component. Server components pass section anchors/titles and storage identity into it; all `window`, `localStorage`, `confirm`, keyboard, and scroll work happens after mount.
- When localStorage is unavailable or corrupted, progress UI remains non-blocking: render the page normally, disable or hide storage-dependent controls as appropriate, and log a console warning.
- Debounced persistence writes the full validated progress object 300ms after updates. Reset writes immediately by removing the current key and clearing in-memory state.
- Collection landing progress is read-only in P0 except for optional reset/bulk controls deferred to P1. Per-section toggles live on lecture pages.
- P1 keyboard shortcuts, resume prompt, milestones, live announcements, print status, bulk collection controls, and doctor/package-review reporting should be implemented after the P0 storage/rendering contract is stable.

## Proposed Implementation Steps

### Step 1: Add a small progress model and storage module
- Create `src/lib/lecture-template/progress.ts` or `src/lib/progress.ts` with pure helpers:
  - `type LectureProgress = Record<string, boolean>`;
  - `type CollectionProgress = Record<string, LectureProgress>`;
  - `singleLectureProgressKey(lectureId: string): string`;
  - `collectionProgressKey(collectionId: string): string`;
  - `sanitizeProgressId(value: string): string`;
  - `validateLectureProgress(value, allowedAnchors): LectureProgress`;
  - `validateCollectionProgress(value, lectures): CollectionProgress`;
  - `calculateLectureProgress(progress, sections): { totalSections, completedSections, percentComplete }`;
  - `calculateCollectionProgress(progress, lectures): { completedSections, totalSections, percentComplete, lectures }`.
- Keep this module free of browser APIs so it is easy to unit test and reuse from server-rendered collection components for deterministic default state.
- Validation should drop unknown section anchors, coerce nothing, require boolean values, and reset malformed root values to empty.

### Step 2: Implement browser storage and `ProgressProvider`
- Create `src/components/lecture-kit/progress/ProgressProvider.tsx` as a client component.
- Expose the spec contract through context: `{ progress, toggleSection, resetProgress, totalSections, completedSections, percentComplete }`.
- Include provider props for `scope`, `storageKey`, `sections`, optional `lectureSlug`, and optional initial collection lecture list.
- On mount:
  - verify `window.localStorage` can be accessed;
  - read the storage key;
  - parse JSON with try/catch;
  - validate against allowed section anchors;
  - set `storageAvailable` false and warn if access fails.
- Debounce writes at 300ms after toggle/bulk changes. Do not write before the initial storage read completes.
- Reset should confirm at the UI boundary, clear state, remove the storage key immediately, and announce "Progress reset."
- Avoid hydration mismatches by rendering 0%/empty progress on the server and updating after mount.

### Step 3: Add lecture progress UI
- Create focused client components under `src/components/lecture-kit/progress/`:
  - `LectureProgressBar.tsx` - progressbar role, summary text, reset button, optional milestones.
  - `SectionCompletionToggle.tsx` - button with `aria-pressed`, accessible label using section title, and completed label/checkmark.
  - `ProgressLiveRegion.tsx` - polite announcements for toggle/reset changes.
  - `ResumePrompt.tsx` - P1 banner that appears for partial progress and jumps to the first incomplete or most recently completed section.
- Update `LecturePage.tsx` to wrap the lecture content with `ProgressProvider`, pass `lecture.sections`, and render `LectureProgressBar` below the header and above `.lecture-layout`.
- Update `SectionRenderer.tsx` to render the completion toggle next to each authored section heading while preserving heading hierarchy and existing section anchors.
- Add a completed class such as `lecture-section-complete` to the section element without dimming body text.

### Step 4: Add collection progress aggregation
- Create `CollectionProgressProvider.tsx` or generalize `ProgressProvider` to support collection storage shape.
- Update `CollectionLanding.tsx` to derive lecture summaries from `validation.results`:
  - use only valid lectures with `result.template` for section totals;
  - keep invalid lectures visible with validation status but no progress percentage or with a "Progress unavailable until valid" label;
  - show overall course percent across all valid lecture sections;
  - show each lecture row's completion percentage and mini progress bar.
- If collection lecture pages are added or already present in a later branch, pass the same collection storage key and `lectureSlug` into `LecturePage` so per-section toggles update the collection object. In this checkout only `src/app/page.tsx` exists under `src/app`, so the implementation should not assume a concrete `src/app/lectures/[slug]/page.tsx` file without first adding or finding it.

## Data Model And Storage Changes

No template schema migration is required. Progress is a runtime browser data model.

```typescript
export interface LectureProgress {
  [sectionAnchor: string]: boolean;
}

export interface CollectionProgress {
  [lectureSlug: string]: LectureProgress;
}
```

Storage keys:

```typescript
const SINGLE_LECTURE_KEY = `lecture-progress:${lectureId}`;
const COLLECTION_KEY = `lecture-progress:collection:${collectionId}`;
```

Validation rules:
- root value must be a non-array object;
- lecture progress values must be booleans;
- keys must match known authored section anchors for the current lecture;
- collection root keys must match known valid lecture slugs;
- malformed JSON or invalid root shape resets to `{}` and logs a warning;
- unknown anchors/slugs can be dropped during validation to handle changed templates.

## API And Component Contract

The provider should expose exactly the P0 contract from the spec for lecture consumers:

```typescript
interface ProgressContextValue {
  progress: Record<string, boolean>;
  toggleSection: (sectionAnchor: string) => void;
  resetProgress: () => void;
  totalSections: number;
  completedSections: number;
  percentComplete: number;
}
```

Implementation may include internal fields such as `storageAvailable`, `loaded`, `setAllSections`, `announce`, or collection summaries, but public components should not depend on raw localStorage.

## Risks, Assumptions, And Open Questions

- Collection lecture routing is not present in this checkout under `src/app`; collection progress on individual lecture pages depends on adding or finding that route before wiring collection-scope toggles outside the landing page.
- localStorage is browser-only, so doctor/package-review commands can report capability and expected keys but cannot report a real user's current completion state from Node.
- Stable `lectureId` and `collectionId` need a product decision. Path-derived ids are predictable for local projects but can change if files move; metadata-derived ids can collide. The implementation should centralize id generation so this can be changed later.
- Debounced writes can race with reset if implemented carelessly. Reset should cancel pending writes before removing storage.
- React server/client boundaries matter: do not import browser-only modules into server components except as client components.
- The resume prompt's "where you left off" requires either last-toggled tracking or an inference. P0 can skip it; P1 should define whether to store a separate last anchor or infer first incomplete section.
- Progress for changed templates should be forgiving: unknown old anchors are dropped, new sections start incomplete, and empty lectures should avoid divide-by-zero.
