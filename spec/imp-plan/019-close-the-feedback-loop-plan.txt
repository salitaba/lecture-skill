Implementation Plan: Close the Feedback Loop — Answer Review, Diagram Inspection, Accessible Shortcuts, and Passage Highlights & Notes (spec/019-close-the-feedback-loop.txt)

Plan status: draft pending tech-lead review

## Brief Summary

`spec/019` closes four loops on top of the now-complete 014-018 work: (P0) two keyboard shortcuts and no skip link are wired but undiscoverable; (P1) quiz/question-set answers are computed and then thrown away on every reload; (P2) 16 authored diagrams are trapped in their authored-width card with no way to inspect or export them; (P3) nothing lets a learner mark up what they're reading — the one genuinely new capability in this spec, not a dropped loop.

The plan is eight steps. Steps 1 (P0) and 4 (P2) are self-contained, low-risk, single-file-family changes. Steps 2-3 (P1) extend `ProgressProvider` with a new, additively-stored `answers` map and a new review component. Steps 5-7 (P3) are the largest and riskiest part of this plan: Step 5 builds a new sibling `AnnotationsProvider` and ships section notes (the fully keyboard-accessible half of P3) on its own, Step 6 adds text-range highlighting (the harder, pointer-first half) as a second, independent slice on top of the same provider, and Step 7 adds the combined review surface. Step 8 is verification. Sequencing notes fully-accessible section notes *before* highlighting deliberately, so P3 ships a complete, accessible feature even if highlighting (the riskiest step in this plan) needs a second pass.

## Corrections And Key Design Decisions

Several things reading the actual code changes *how* to implement what the spec describes, without changing what it asks for:

- **"The same per-lecture storage key... not a new storage scheme" (spec Product Decisions, R1 and H1-H3) cannot mean the literal same `localStorage` value.** `LectureProgress` is `Record<string, boolean>` (`src/lib/lecture-template/progress.ts:3`) with a narrow validator, `validateLectureProgress` (lines 59-71), that already discards any non-boolean value on read. Writing quiz-attempt objects or highlight arrays into that same value would require migrating the shape of data that's already live in every returning learner's browser. This plan instead adds two **sibling** keys derived from the same `lectureId` via two new functions in `progress.ts` — `singleLectureAnswersKey(lectureId)` → `lecture-progress:{id}:answers` and `singleLectureAnnotationsKey(lectureId)` → `lecture-progress:{id}:annotations` — reusing `singleLectureProgressKey`'s exact naming family and, in `ProgressProvider`/the new providers, the exact same debounced-write / try-catch-read / `storageAvailable`-fallback mechanics already proven at `ProgressProvider.tsx:76-129`. This satisfies the spec's actual intent (one mechanism, one id, no new storage system) without a breaking migration of existing section-completion data.
- **`Quiz` and `QuestionSet` must consume the new answer state through an *optional* context hook, not a required one.** `tests/lecture-template/quiz-interaction.test.tsx` and `tests/lecture-template/assessment-interaction.test.tsx` (confirmed by reading both) render `<Quiz>`/`<QuestionSet>` directly, with no wrapping provider. `ProgressProvider.tsx:358-360` already has exactly this shape solved once — `useProgressOptional()` returns `undefined` outside a provider instead of throwing, unlike `useProgress()` (lines 352-356). Step 2 extends `ProgressContextValue` and has `Quiz`/`QuestionSet` call `useProgressOptional()`, so every existing standalone-render test keeps passing unchanged, and restore/persist behavior is simply a no-op when nothing wraps them.
- **P3's new state does not belong in `ProgressProvider`.** `ProgressProvider.tsx` is already 360+ lines covering section toggling, resume targeting, toasts, collection sync, and the two keyboard shortcuts — none of which highlights or notes need to interact with (a note doesn't affect `percentComplete`, doesn't need a resume target, doesn't fire a toast). `CollectionProgressProvider.tsx` is existing precedent that a second, small, independent provider living in `progress/` alongside `ProgressProvider` is a normal pattern in this codebase, not a deviation from it. Step 5 adds `progress/AnnotationsProvider.tsx` as a third such provider, mirroring `ProgressProvider`'s persistence mechanics but owning only `{ highlights, notes }`. (Quiz-attempt state (R1) is small and lifecycle-bound to the same per-lecture session as section completion, so it *does* go into `ProgressProvider` — see Step 2 — this split is deliberate, not inconsistent.)
- **Diagram fullscreen (D1) needs the rendered SVG string kept in React state, not just injected via `innerHTML`.** `Diagram.tsx:44-46` currently sets `containerRef.current.innerHTML = svg` directly and discards `svg` otherwise. Step 4 adds `const [svgMarkup, setSvgMarkup] = useState<string | null>(null)` alongside that existing assignment so the same markup can be reused for the dialog's enlarged copy and for the download `Blob`, without touching the existing ref-based inline render path (lower risk than switching the inline render to `dangerouslySetInnerHTML`).
- **`Icon.tsx`'s `IconName` union (`"chevron" | "check" | "warning" | "arrow-prev" | "arrow-next" | "sun" | "moon"`) has no glyph for expand, download, or note-present.** Adding new entries to `paths`/`IconName` (Icon.tsx:1-11) is a small, additive change to an existing primitive, not a new component — consistent with the spec's non-goal on new component-kit primitives. Steps 4 and 5 each add one glyph.
- **Text-range highlighting (H1) is DOM-manipulation work, not React-tree work, and is the highest-risk step in this plan.** Authored section content renders through `renderMarkdownBlocks`/`RenderBlocks` into ordinary React-owned DOM; there is no existing raw-HTML-string representation of a section's prose to slice by character offset (unlike `Diagram.tsx`'s Mermaid SVG, which *is* a raw string). Applying a saved highlight means, after React commits, walking a block element's text nodes to the saved offsets and wrapping them with a real `<mark>` via `Range.surroundContents()` — which throws if the range's boundaries fall inside a nested inline element (e.g. a `<strong>` partially inside the selection) and needs a documented fallback. See Step 6 and Risks.
- **`SectionProgressFrame` (`progress/SectionProgressFrame.tsx`) is the existing per-section DOM anchor (`id={anchor}` at line 18) but wraps the heading row and completion toggle too, not just prose.** Step 6 does not attach selection listeners there; it wraps only `RenderBlocks`'s output in a new `HighlightableContent` component (see Step 6), so selecting text in a section heading or the note textarea can never trigger a highlight prompt.

### Step 2: Persist and restore quiz/question-set attempts (spec P1, R1-R2)

- In `progress.ts`, add `singleLectureAnswersKey(lectureId)` returning `lecture-progress:{lectureId}:answers`, alongside `singleLectureProgressKey` (line 30). Add a minimal `AnswerAttempt = { selected: string; correct: boolean }` type and `AnswerAttempts = Record<string, AnswerAttempt>` and a narrow `validateAnswerAttempts(value, allowedKeys?)` validator mirroring `validateLectureProgress` (59-71) — note allowed-key validation here can only filter by *shape* (string/boolean), not a known-anchor allowlist, since the caller doesn't have every possible item key upfront the way section anchors are known; document this as an intentional, narrower validation than `validateLectureProgress`'s.
- In `ProgressProvider.tsx`, add `answers` state, load it in a second effect parallel to the existing load effect (76-108) reading `singleLectureAnswersKey(...)` (the provider needs a `lectureId`, not just `storageKey`, as a new prop — or derive the answers key from the existing `storageKey` prop via a small string transform if `lectureId` isn't already passed in; confirm which is cleaner against `LecturePage.tsx`'s existing `ProgressProvider` call site (47-52) during implementation), write it with the same debounce pattern (110-129) under its own timer ref, and expose `answers: AnswerAttempts` and `recordAnswer(key: string, selected: string, correct: boolean): void` on `ProgressContextValue` (15-32).

- Run after this step: `npm run test -- tests/lecture-template/quiz-interaction.test.tsx tests/lecture-template/assessment-interaction.test.tsx tests/lecture-template/progress-provider.test.tsx tests/lecture-template/progress-model.test.ts`.

### Step 6: Text-range highlighting (spec P3, G8, H1 — the pointer-first, higher-risk half)

- New `src/components/lecture-kit/HighlightableContent.tsx`, `"use client"`, wrapping `RenderBlocks`'s output (replacing the direct `<RenderBlocks .../>` call at `SectionRenderer.tsx:39` with `<HighlightableContent anchor={section.anchor} blocks={section.blocks}>...</HighlightableContent>` or equivalent — exact prop shape decided during implementation, but it must own a `ref` to a wrapping `<div>` scoped *only* to the rendered blocks, not the heading/toggle/note controls, per the Corrections note on `SectionProgressFrame`).
- On `mouseup`/`selectionchange` inside that ref, if `window.getSelection()` is non-collapsed and fully contained within one child block element: compute the block's index among its siblings, and the start/end character offsets within that block's `textContent` (via `Range.cloneRange()` + `setStart(blockElement, 0)` + `.toString().length`, a standard offset-computation technique — cite in code comment, not a library). Show a small floating `Button` ("Highlight") positioned via the selection's `getBoundingClientRect()`.
- On click, call `addHighlight({ sectionAnchor, blockIndex, start, end, snippet: selection.toString().slice(0, 80) })`, then `window.getSelection()?.removeAllRanges()`.
- On mount and whenever `highlights` changes, re-apply saved highlights for this section by walking the block element at each `blockIndex`, constructing a `Range` at the saved `start`/`end` via a text-node-walking offset lookup (the inverse of the capture step), and wrapping it with `document.createElement("mark")` via `range.surroundContents(mark)`. If `surroundContents` throws (range crosses an inline-element boundary) or the block's current `textContent.slice(start, end)` no longer equals the saved `snippet`, drop that highlight silently for this render (per spec's Product Decision and the Corrections note) rather than retrying with a more complex extract/reinsert fallback in this first pass — log a `console.warn` in dev only.
- Clicking an already-highlighted `<mark>` shows a small "Remove highlight" `Button`; click calls `removeHighlight(index)` (or a stable highlight id if an index proves unstable across re-renders — decide during implementation; a content-hash-based id keyed on `sectionAnchor:blockIndex:start:end` is safer than array index and is cheap to compute).
- `<mark>` styling: background uses the existing `--accent` token at reduced opacity (a new CSS custom property `--highlight-bg: color-mix(in srgb, var(--accent) 25%, transparent)` or an equivalent fixed rgba, decided during implementation against actual token values) — no new semantic color per the spec's Non-Goals.
- Add the floating highlight button and `<mark>` transitions to the `prefers-reduced-motion` block.
- Run after this step: new tests only (see Test Plan) — this step has no prior test coverage to preserve, since the feature doesn't exist today.

## Data Model, API, Persistence, Event, Or Domain Changes

- `progress.ts`: three new pure functions (`singleLectureAnswersKey`, `singleLectureAnnotationsKey`, plus their validators) and three new types (`AnswerAttempt`/`AnswerAttempts`, `Highlight`, `Annotations`). `LectureProgress`'s existing shape and `singleLectureProgressKey`/`collectionProgressKey` are untouched.
- Three `localStorage` keys per lecture instead of one: `lecture-progress:{id}` (existing, untouched), `lecture-progress:{id}:answers` (new, Step 2), `lecture-progress:{id}:annotations` (new, Step 5). No collection-level equivalent for either new key — both are per-lecture only, per spec.
- `ProgressContextValue` (`ProgressProvider.tsx:15-32`) gains `answers`/`recordAnswer`.
- New context/provider: `AnnotationsProvider`/`useAnnotations`/`useAnnotationsOptional`.
- New components: `AnswerReviewDisclosure.tsx`, `HighlightableContent.tsx`, `AnnotationsIndexDisclosure.tsx`. `SectionRenderer.tsx` and `LearningPathNavigation.tsx` get small additive changes, not rewrites.
- `Icon.tsx` gains three glyphs: `expand`, `download`, `note`.
- `globals.css` gains: `.skip-link`, shortcut-hint text styling, `.diagram-dialog*`, `<mark>`/highlight styling, note-control and note-indicator styling — all additive; no existing rule is removed except where Step 4 confirms native `<dialog>` default styling already handles print-hiding (verify before deleting anything).
- No authoring schema, parser, validator, or `LectureTemplate`/`LectureComponent` type changes anywhere in this plan.

## Risks, Assumptions, And Open Questions

- **Step 6 (text highlighting) is materially riskier than everything else in this plan and should be reviewed as its own change, potentially by someone with recent `Range`/`Selection` API experience**, before being bundled into the same release as Steps 1-5/7. If it proves too fragile in practice (e.g., jsdom can't meaningfully exercise it, or `surroundContents` failures are common enough on real authored prose with lots of inline emphasis/links/glossary terms), consider shipping Steps 1-5 and 7 first and revisiting Step 6 as its own follow-up rather than blocking the rest of spec 019 on it.
- **Whether `ProgressProvider` should take a new `lectureId` prop or derive the answers/annotations keys from its existing `storageKey` prop is left open** (Step 2) — resolve by checking whichever is less invasive at `LecturePage.tsx`'s actual call site during implementation, not assumed here.
- **The exact CSS value for the highlight background (`--highlight-bg`) depends on values in `globals.css`'s current token block that should be re-read at implementation time**, not assumed from this plan (tokens may have shifted since this plan's research pass).
- **Highlight identity (index vs. content-hash id) for removal (Step 6) is flagged but not fully resolved** — recommend a hash-based id over array index for stability across re-renders/re-ordering, decide firmly before writing `removeHighlight`.
- **No visual-regression tooling exists in this repo** (same constraint noted in 016/017/018's plans) — all before/after comparisons are manual screenshots at 390/768/1280px, light and dark.
- **`globals.css` line numbers cited throughout this plan (especially the `prefers-reduced-motion` block and print block) were read during this plan's research pass and may drift** if other work lands first — re-grep before editing rather than trusting line numbers verbatim, same caution 018's plan flagged for its own citations.
- **Question-set sub-question review links (Step 3) point at the question-set's shared anchor, not an individual sub-question**, since sub-questions have no DOM id today — a learner reviewing a missed sub-question lands on the right card but must find the specific question themselves. Adding individual sub-question anchors would touch `QuestionSet.tsx`'s render output; judged out of proportion for this plan but flagged as a small, known limitation of Step 3's implementation, not the spec's requirement.
