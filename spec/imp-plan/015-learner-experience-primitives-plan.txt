Implementation Plan: Learner Experience Primitives (spec/015-learner-experience-primitives.txt)

Plan status: draft pending tech-lead review

## Brief Summary

`spec/015` is a sibling of `spec/014-senior-ui-ux-polish.txt`. Spec 014 already shipped the four P0 page-level primitives (CollectionPrimaryAction, LectureNavigation wiring, active section state, SelectableQuiz) and a number of supporting P1 pieces. Spec 015 is self-contained for the new work and consolidates the remaining P1 cross-cutting helpers plus a small amount of P0 polish that the spec explicitly defers:

- B1. `ResumePrompt` as a real per-session helper, placed exactly where the spec wants it, not inside the progress bar.
- B2. `ReadingTimeEstimate` derived at render time from authored section word counts.
- B3. `GlossaryIndex` auto-generated from authored `glossary_term` components, with a per-lecture and whole-collection surface.
- B4. `AssessmentIndexDisclosure` as a dedicated, count-aware disclosure component that can be placed under the lecture list and that wraps the existing assessment index list.

The plan preserves the existing authoring schema, parser, validator, local-only progress model, static export, review package, no-account behavior, and print-friendly output. It introduces no new `lecture-component` type, no new frontmatter field, no new dependency, and no breaking change. It is a combination of a small new lib helper for reading time, a few new components, refinement of the existing 014 ResumePrompt, and focused CSS plus tests.

`spec/014` already produced, in this checkout:

- `src/components/lecture-kit/CollectionPrimaryAction.tsx` plus `src/lib/lecture-template/resumeTarget.ts` (P0-A1).

This plan therefore does not redo 014 work. It tightens P1-B1 against the 015 spec, then adds P1-B2, P1-B3, and P1-B4 as fresh work. Anything 014 already defined for the P0 list is referenced rather than re-planned.

## Proposed Implementation Steps

### Step 1: Add a pure `readingTime` helper

- Add `src/lib/lecture-template/readingTime.ts` exporting:
  - `countWordsInRenderBlock(block: RenderBlock): number`
  - `countWordsInSection(section: LectureSection): number`
  - `countWordsInLecture(lecture: LectureTemplate): number`
  - `estimateReadingMinutes(totalWords: number, wpm = 200): number` - rounds to the nearest minute, with a 1-minute floor, mirroring the spec text.
  - `lectureReadingMinutes(lecture: LectureTemplate): number` - composes the above.
  - `sectionReadingMinutes(section: LectureSection): number` - same composition, for the optional per-section estimate in the learning path.
- Counting rules:
  - Count only authored section text, not overview, objectives, takeaways, or the answer key appendix. The spec describes "per-lecture reading time, derived from authored section word counts" and the preferred flow keeps the reading estimate next to the lesson body.
  - For `paragraph`, `bullet_list`, `numbered_list`, and `table` markdown blocks, count whitespace-separated tokens.
  - For `code_fence` blocks, count code tokens.
  - For `heading` blocks, count heading text tokens.
  - For component blocks, count words in user-visible text fields only: `body`, `definition`, `context`, `prompt`, `scenario`, `task`, `question`, `walkthrough`, `solution`, `takeaway`, `mistake`, `why_it_fails`, `correction`, `example_before`, `example_after`, `quote`, `attribution`, `content`, `items[].body`, `items[].detail`, `items[].label`, `steps[]`, `hints[]`, `items[]` (for checklist), `items[].prompt` / `items[].answer` for flashcard prompts/answers, etc. Do not include YAML keys, anchor strings, or answer keys (the answer key is already an appendix). Keep the helper simple: skip component blocks whose text is fully internal to the assessment feedback layer (`quiz.answer`, `quiz.explanation`, `practice_task.rubric.expected`, `free_response.guidance`, `question_set.questions[].answer`, `question_set.questions[].feedback`) to avoid double counting with the appendix and to keep the estimate honest.
  - Aliases should not be counted as separate words; they belong to the term and are short.
- Add `tests/lecture-template/reading-time.test.ts` with cases for:
  - empty lecture, single short paragraph, mixed blocks and components;
  - rounding at 0.5 minute boundaries;
  - one-minute floor for non-zero lectures;
  - the spec example: a 2000-word lecture yields `~10 min read` (200 wpm, round to nearest minute).
- The helper must be pure and deterministic; the same lecture must yield the same value at build time and during review-package assembly.

### Step 3: Add `GlossaryIndex` and a collection-level glossary surface

- Add `src/lib/lecture-template/glossaryIndex.ts` exporting:
  - `GlossaryEntry { term: string; definition: string; context?: string; aliases: string[]; lectureSlug?: string; sectionAnchor: string; }`
  - `collectLectureGlossary(lecture: LectureTemplate, lectureSlug?: string): GlossaryEntry[]`
  - `collectCollectionGlossary(validation: CollectionValidationResult): GlossaryEntry[]`
  - `dedupeAndSort(entries: GlossaryEntry[]): GlossaryEntry[]` - sorts alphabetically by `term.toLowerCase()`, folds aliases into a single entry when the same term appears multiple times, prefers the first non-empty definition, and keeps all source lecture/section anchors for cross-lecture references.
- Add `src/components/lecture-kit/GlossaryIndex.tsx` (server component):
  - Props: `entries: GlossaryEntry[]`, `id?: string`, `title?: string` (default `Glossary`), `scope?: "lecture" | "collection"` (default `lecture`).
  - If `entries` is empty, render `null`. The spec says authors do not maintain the index; an empty index should be invisible.
  - Renders a `<section className="glossary-index lecture-panel" aria-labelledby={headingId}>` containing:
    - a `<details>` (the surface) with `<summary>Glossary · N terms</summary>` on mobile (default closed) and an expanded `<ol class="glossary-index-list">` on desktop using CSS only. The `details` is the print-safe default; desktop overrides via CSS to start expanded. The same node serves both viewports.
    - The list uses semantic markup. Each item is `<li class="glossary-index-item"><a class="glossary-index-term" href="#{anchor}">{term}</a><p class="glossary-index-definition">{definition}</p></li>` where the link target is the section anchor on the lecture page. For collection-scope entries that appear in multiple lectures, render one link per occurrence with the lecture title as supplementary text.
    - Term is bold, definition is muted and one-line truncated in CSS with `text-overflow: ellipsis` and the full text in `title`.
- Add `src/components/lecture-kit/CollectionGlossaryIndex.tsx` that wraps `GlossaryIndex` with `scope="collection"` and an `aria-label="Course glossary"`.
- Add tests:
  - `tests/lecture-template/glossary-index.test.tsx` - renders nothing for a lecture with no `glossary_term` components; renders one entry per term for a lecture with multiple; sorts alphabetically; folds aliases into the entry; renders the correct anchor.
  - `tests/lecture-template/collection-glossary.test.tsx` - aggregates across multiple valid lectures; the same term in two lectures yields two anchors; an invalid lecture does not contribute.
  - `tests/lecture-template/glossary-index-helper.test.ts` - pure helper tests for `collectLectureGlossary` and `collectCollectionGlossary`, including alias folding and ordering.
- Update `src/app/globals.css`:
  - `.glossary-index .glossary-index-list` uses a three-column grid on wide viewports and stacks on narrow.
  - `.glossary-index summary` shows the count and chevron using the same chevron pattern as the learning-path mobile disclosure.
  - Print rule: include the glossary index in the printed handout after the lecture body, before the answer key. On the collection landing, the collection glossary index appears below the assessment index disclosure.

### Step 6: Tighten `ResumePrompt` against the 015 spec

- The current `src/components/lecture-kit/progress/ResumePrompt.tsx` is too wide (it includes a `Dismiss` button) and is placed inside the progress bar. The 015 spec wants:
  - "Compact banner directly below the lecture header, above progress and the learning-path nav."
  - "Single primary button, no card border; uses a divider row."
  - "Dismissal is per-session (sessionStorage), not persistent. A reload re-shows the prompt if the section is still incomplete."
  - "Hidden on print." (Already true via `.resume-progress-prompt { display: none }` in print CSS, but verify.)
- Move the render call:
  - Update `src/components/lecture-kit/progress/LectureProgressBar.tsx` to remove the `<ResumePrompt />` import and render call. The progress bar's footer now only contains the `Reset progress` button.
  - Update `src/components/lecture-kit/LecturePage.tsx` to render `<ResumePrompt />` directly under the `LectureHeader` and `<LectureProgressBar />` block but above the `CollectionTopNav` and the `lecture-layout` div, so the prompt sits between progress and the learning path. The spec asks for "below the lecture header, above progress and the learning-path nav"; the closest accurate placement is between the progress bar and the `CollectionTopNav` because the progress bar is part of the header band today, but a stricter reading places the prompt above the progress bar. Implementers should consult the spec UX notes and place it immediately after the `LectureHeader` and before the `LectureProgressBar`, so the prompt appears before the progress bar. Update the test to assert this exact order: `LectureHeader > ResumePrompt > LectureProgressBar > CollectionTopNav > lecture-layout > LectureNavigation`.
- Change the markup:
  - Replace the card-style `<div className="resume-progress-prompt" role="status">` with a divider-row `<aside className="resume-prompt" aria-label="Resume lecture">` (or `<section>` per the spec, but `<aside>` is the better semantic role for a tangential affordance). The prompt uses a row layout with the message and a single primary link/button.
  - Drop the `Dismiss` button entirely. The spec's "dismissible per session" is implemented by clicking anywhere on the prompt (the message and the button are part of the same `details`-like disclosure), but a simpler model is to keep one button: the button label is `Continue from <Section title>` and the action jumps to the section. The spec says "Single line: `Continue from <Section title>` plus a primary action." A secondary "Dismiss" button is not desired. Dismissing therefore happens implicitly on click (which also navigates). For accessibility, add a separate, but compact, secondary `Dismiss for this session` link button at the end of the row, styled as plain text. The 015 spec prioritizes the single primary action; the dismiss is allowed when it does not crowd the prompt. If implementation can keep just the primary, do so; the fallback is a `Dismiss` link.
  - Use a real `<a href="#{section.anchor}">` for the primary action so the prompt works without JavaScript. The `<a>` is the primary, the dismiss is a button.
- Change the dismiss path:
  - Replace the `useState`-backed `resumeDismissed` flag in `ProgressProvider` with a per-session flag stored in `sessionStorage` under a stable key, e.g. `lecture-resume-dismissed:{storageKey}`. The provider still exposes `dismissResumePrompt()` and `resumeTarget`; only the backing store changes.
  - On mount, read the session flag. On `dismissResumePrompt()`, write `1` to that key. A page reload reads the same key, so the prompt re-shows when the section is still incomplete (per spec). When the user closes the tab, the key is cleared by the browser.
  - When `resumeTarget` becomes undefined (lecture 100% complete, or all sections complete), the dismiss flag is effectively dead - the prompt is not rendered. The next visit to the lecture should still re-show the prompt for a new section, which sessionStorage guarantees because the key is the same for the lecture.
- Update CSS:
  - Add `.resume-prompt` divider-row styles (border-top, border-bottom, padding, 44px min-height on touch).
  - Hide `.resume-prompt` in `@media print` (already true for `.resume-progress-prompt`; rename the class and add a guard so the new class is also hidden).
  - Honor `prefers-reduced-motion: reduce` (no transitions).
- Update tests:
  - Add a test in `tests/lecture-template/resume-prompt.test.tsx` that verifies the prompt renders between the header and the progress bar, contains exactly one primary link, and that clicking the link (or pressing Enter on it) jumps to the section and dismisses the prompt for the rest of the session.
  - Add a test for `sessionStorage` behavior: dismissing writes to `sessionStorage`; a remount within the same session does not re-show the prompt; the prompt becomes visible again when the section is still incomplete and the page reloads (simulate by clearing React state and reloading the same storage).
  - Update `tests/lecture-template/lecture-components.test.tsx` to reflect the new class name and the new placement, including the order assertion.
  - Update `tests/lecture-template/visual-hierarchy.test.tsx` if it asserts specific class names.

## Data Model, API, Persistence, Event, Or Domain Changes

- Template schema: no changes.
- Parser, validator, renderMarkdown: no changes expected. The reading-time helper is a pure function over the already-parsed `RenderBlock` tree.
- Persistence: one new `sessionStorage` key per lecture to back the resume-prompt dismiss flag, scoped as `lecture-resume-dismissed:{storageKey}`. No new `localStorage` key. The review package and `npm run package:review` static export are unaffected because sessionStorage is per browser session only and is not serialized into the static build.
- Component API changes:
  - `LectureHeader` adds an optional `readingMinutes` prop.
  - `LearningPathNavigation` adds an optional `sectionMinutes` map.
  - `CollectionLanding` no longer renders the assessment disclosure inline; it delegates to `AssessmentIndexDisclosure`.
  - `LecturePage` renders `ResumePrompt` directly (no longer inside `LectureProgressBar`).
  - `ProgressProvider` swaps the in-memory `resumeDismissed` state for a `sessionStorage`-backed flag with the same external contract.
- External services: none.
- Runtime network dependencies: none.
- Static export and review package behavior: must remain compatible. Reading time and glossary index are derived at render time and serialized into the static HTML. The review-package worksheet must list the same values.

## Risks, Assumptions, And Open Questions

- The exact algorithm for "authored section word counts" is not specified for component blocks. The plan above proposes a conservative counting rule that ignores assessment feedback text to avoid double counting with the answer key appendix. Implementers should call this out in the PR description and update tests if a different rule is preferred.
- The reading-time helper is render-time, not build-time. Next.js static export (`npm run build`) will bake the value into the static HTML, so the static export and the review package will match. If a future build pipeline wants to compute reading time at build time and inject it as a constant, that is a non-goal for 015 and can be deferred.
- The new `sessionStorage` flag means the resume prompt behaves differently between a page reload (re-shows) and a tab close/reopen (re-shows, since `sessionStorage` is per tab session). This matches the spec. A subtle consequence is that the prompt is not "remembered" across tabs; the spec accepts this.
- The "Default open on desktop" plan for the assessment disclosure uses a CSS approach. If the implementer prefers a server-rendered `open` attribute and a CSS rule that closes it on mobile, the tests must not assert the `open` attribute on the server in a way that conflicts with the responsive default. Pick one path and pin it in tests.
- The per-section minutes in the learning path can crowd the section label on long titles. The spec says "Optional per-section estimate in the learning-path nav (e.g. `3 min`), if it does not crowd the section label." The CSS rule should ellipsize the label first and never the minutes.
- A glossary term that appears in two different lectures with different definitions is rare but possible. The plan folds them into one row with both anchors. If the same term has two different definitions, prefer the first and note the second in `title`; this is documented in the helper test.
- The spec lists "Above any instructor notes" in the assessment index disclosure description, but instructor notes are authored inside lectures, not on the collection landing. The collection landing therefore renders the assessment disclosure below the lecture list without an instructor notes band. The spec line about instructor notes applies if a future collection-level instructor note is added; the plan handles the current state and is forward-compatible.
- Browser screenshot verification may need a dev server. If the repo does not already include Playwright, prefer manual browser checks or a lightweight script rather than adding a heavy dependency.
- Existing 014 tests use the class `.resume-progress-prompt`. After B1, the new class is `.resume-prompt` (or kept as the old class for compatibility - decide and document). The CSS migration is part of Step 6.
- The spec is a sibling of 014 and explicitly defers P0 items to 014. If a previous 014 task did not land (for example, the `CollectionPrimaryAction` may need an `aria-label` that includes the target lecture title per 015 A1 accessibility rules), the implementer should fold that small refinement into the relevant B1-B4 step rather than creating a separate task. The plan does not list 014 follow-ups as separate tasks; this is a deliberate choice to keep tasks small and the spec 015 self-contained.
