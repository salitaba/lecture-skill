# Implementation Plan: Learning Loop — Dashboard, Objective Evidence, and Spaced Review

## Brief Summary

Implement `spec/025-learning-loop-mastery-and-review.txt` as a local-first
learning loop on top of the existing Lecture Site Engine. The change adds
stable objective records, objective-to-assessment validation and evidence,
learner dashboard summaries, a deterministic local spaced-review scheduler, and
review-queue controls. It must preserve existing lecture rendering, answer
attempts, section progress, annotations, static export, print output, and
no-JavaScript behavior.

This plan does not implement accounts, sync, analytics, grading, search, AI
tutoring, social features, or a new authored assessment component.

## Relevant Existing Architecture

- `src/lib/lecture-template/types.ts` owns parsed and normalized lecture,
  assessment, objective, validation, and collection contracts.
- `src/lib/lecture-template/parseTemplate.ts` parses Markdown headings and list
  blocks, including the `## Learning Objectives` list.
- `src/lib/lecture-template/validateTemplate.ts` normalizes objectives and
  validates assessment `id` and `objective_refs` fields.
- `src/lib/lecture-template/assessments.ts` creates deterministic assessment
  summaries, collection assessment indexes, answer definitions, and diagnostics.
- `src/lib/lecture-template/progress.ts` owns local progress, answer attempts,
  annotations, storage keys, and corrupted-state validation.
- `src/components/lecture-kit/progress/ProgressProvider.tsx` persists lecture
  progress and objective-choice answer attempts in browser `localStorage`.
- `src/components/lecture-kit/progress/CollectionProgressProvider.tsx` owns
  collection-level progress aggregation and resume state.
- `src/components/lecture-kit/CollectionLanding.tsx` renders course facts,
  primary resume action, progress, lecture list, assessment index, glossary,
  and reviewer status.
- `src/components/lecture-kit/LecturePage.tsx` composes lecture progress,
  annotations, assessment index, objectives, sections, glossary, answer review,
  and navigation.
- `src/components/lecture-kit/assessment/AssessmentShell.tsx` provides shared
  choice and local self-assessment lifecycle behavior.
- `src/components/lecture-kit/Quiz.tsx`, `QuestionSet.tsx`,
  `Flashcard.tsx`, `FreeResponse.tsx`, and `PracticeTask.tsx` are the existing
  assessment interaction surfaces to extend with review controls.
- `src/lib/lecture-template/reviewPackage.ts`, `doctor.ts`, and
  `validateCli.ts` define reviewer-facing and machine-readable diagnostics.
- Existing tests under `tests/lecture-template/` use Vitest, Testing Library,
  local fixtures, and focused provider/component assertions.

## Proposed Data And API Changes

### 1. Normalize learning objectives

Update `src/lib/lecture-template/types.ts` with a `LearningObjective` model
containing `id`, `text`, `isExplicit`, `idLocator`, and `textLocator`. Change
`LectureTemplate.objectives` from `string[]` to the structured representation
while keeping rendering backward-compatible through a text field.

Update `parseTemplate.ts` and `validateTemplate.ts` so Markdown objective list
items support:

```text
- [build-timeline] Build an incident timeline that separates evidence from conclusions.
```

The parser must retain per-item list-line locators rather than only the
`bullet_list` block locator. Use the anchored grammar
`^\[([a-z0-9]+(?:-[a-z0-9]+)*)\]\s+(.+?)\s*$` and reuse
`isAnchorSafeAssessmentId()` from `src/lib/lecture-template/anchors.ts`.
Reject malformed bracket-prefixed items, invalid IDs, empty IDs, and markers
with no text. Preserve ordinary unmarked items as legacy objectives. Generate
collision-safe, ordered, display-only IDs for legacy items, mark them
`isExplicit: false`, and never resolve `objective_refs` to them. Duplicate
explicit IDs and explicit/generated collisions must produce author-locatable
diagnostics. Add tests for invalid markers, duplicate text, reordered legacy
items, and explicit/generated collisions.

Update all objective rendering and test fixtures that currently assume a
`string[]`. Do not modify raw-source evidence files.

### 2. Add pure evidence and review models

Create focused pure modules, preferably:

- `src/lib/lecture-template/objectiveEvidence.ts`
- `src/lib/lecture-template/reviewSchedule.ts`

The exact filenames may follow existing naming conventions, but the modules
must not import React or browser APIs.

`objectiveEvidence.ts` should:

- accept normalized objectives, assessment summaries, and a normalized
  `LearnerActivityEvidence` adapter containing assessment key, response key,
  attempted/revealed state, objective outcome, self-assessment, review record,
  and due state;
- resolve linked assessments by objective ID;
- derive `not_started`, `practicing`, `review_recommended`, `demonstrated`, and
  `not_assessed` statuses;
- use exact precedence: any incorrect/needs-review/due activity means
  `review_recommended`; otherwise any unattempted or incomplete activity means
  `practicing`; only all-positive completed activities are `demonstrated`;
- treat an objective with no linked activity as `not_assessed`;
- return links to the source lecture, section anchor, and assessment anchor;
- avoid percentages, grades, and hidden answer text.

For `question_set`, keep one authored assessment summary but create one
schedulable activity per question, keyed as `${assessmentId}:${questionIndex}`.
Mixed question outcomes use the same precedence rules. A due item counts as
`review_recommended` even when its previous rating was positive.

`reviewSchedule.ts` should:

- define review ratings and persisted review state. Persist `status` as
  `learning` when repetitions are zero and `review` otherwise; derive due state
  from `dueAt` rather than persisting a second, drift-prone due flag;
- implement the bounded default interval sequence `0, 1, 3, 7, 14, 30` days;
- calculate the next state deterministically for `again`, `hard`, `good`, and
  `easy` ratings using an injected valid ISO `now` value;
- use `again => 0`, `hard => ceil(previous * 1.5)` with a minimum of one,
  `good => next sequence interval`, and `easy => one additional sequence step`,
  with a 30-day cap;
- reject invalid `now` inputs, discard invalid persisted dates before calling
  the scheduler, and clamp clock-rollback transitions to the later of `now`
  and `lastReviewedAt`;
- expose due-state and queue-order helpers;
- contain no storage or UI code.

Add a transition-table test for new and existing records, all ratings, zero-
interval behavior, the 30-day cap, exact ISO arithmetic, invalid `now`, and
clock rollback.

The transition contract is:

| Existing state | Rating | Repetitions | Interval | Status |
| --- | --- | ---: | ---: | --- |
| none | again | 0 | 0 | learning |
| none | hard/good | 1 | 1 | review |
| none | easy | 1 | 3 | review |
| any | again | 0 | 0 | learning |
| any | hard | max(1, previous) | max(1, ceil(previous × 1.5)) | review |
| any | good | previous + 1 | next `[1, 3, 7, 14, 30]` interval | review |
| any | easy | previous + 1 | one additional sequence step, capped at 30 | review |

The scheduler receives an injected valid ISO `now`; invalid `now` throws a
documented `RangeError`. Persisted invalid records are pruned before this
function runs. Clock rollback uses the later of `now` and `lastReviewedAt` for
the next transition, while `isDue(record, now)` remains false until `now`
reaches `dueAt`.

### 3. Extend validated collection summaries

Extend `src/lib/lecture-template/assessments.ts` or a neighboring pure module
to expose objective-aware assessment summaries without copying answer text into
learner-facing records. Add collection helpers that preserve lecture slug,
section anchor, objective references, and stable assessment IDs.

Extend validation and review-package models to report objective coverage:

- total objectives;
- explicit-ID count;
- assessments with unresolved references;
- objectives with no linked assessment;
- assessment counts by evaluation mode;
- `learnerStateIncluded: false`.

Define a serializable `ObjectiveDiagnostics` contract and propagate it through
single-lecture validation JSON, collection validation JSON, the review-package
lecture record, `MANIFEST.md`, `manifest.json`, and reviewer disclosures. Keep
learner attempts, review records, due dates, and answer text excluded. Coverage
diagnostics belong in author/reviewer surfaces, not the learner dashboard.

Keep answer-key behavior unchanged.

### 4. Persist review state locally

Extend the learner-state boundary without merging unrelated data structures:

- add a namespaced review key derived from `lectureId`, such as
  `lecture-progress:{lectureId}:reviews`;
- add validation and stale-record pruning in `progress.ts` or a focused review
  state module;
- add a client provider/hook for reading and writing review state, preferably
  separate from the existing section-progress provider unless composition is
  demonstrably simpler;
- make writes debounced and best-effort like existing progress and annotation
  persistence;
- expose a loaded/storage-available state so server render and hydration never
  claim that an item is due before local data is loaded;
- keep unknown assessment IDs and malformed records out of runtime state.

When localStorage is unavailable, keep review actions in memory for the current
session and expose an explicit non-persistence state. Review loading and
storage availability are independent from existing answer loading. The existing
section `resetProgress()` does not clear answers, annotations, or reviews; add a
separate review reset action.

Existing answer-attempt keys and section-progress keys must remain readable.
The implementation must retain `singleLectureAnswersKey()`,
`AnswerReviewDisclosure`, and `AnswerKeyAppendix` behavior from specs 019 and
024. Only scheduler metadata for non-choice activities is persisted; no
free-response text, drafts, or full mounted lifecycle state is stored.

### 5. Add learner surfaces

Create focused components, reusing existing `Card`, `Button`, `Disclosure`,
`ProgressMeter`, and assessment primitives:

- `src/components/lecture-kit/LearnerDashboardSummary.tsx`
- `src/components/lecture-kit/ObjectiveEvidenceMap.tsx`
- `src/components/lecture-kit/ReviewQueue.tsx`
- `src/components/lecture-kit/AssessmentReviewControls.tsx`

Integrate them into:

- `CollectionLanding.tsx` for collection-level continue/review/objective
  summaries;
- `LecturePage.tsx` for lecture-level objective and review summaries;
- `AssessmentShell.tsx` and the existing assessment components for review
  ratings and retry behavior.

The queue should link to the authored source assessment anchor rather than
duplicating answer-bearing assessment markup. Collection queues may group by
lecture but must preserve authored order within a lecture.

Add a collection review registry containing `{ slug, lectureId, order,
objectives, assessments }`. Derive `lectureId` from each valid template path
using the existing progress ID helper. Read each lecture's review key in the
collection provider or a composed review snapshot aggregator. Namespace
objective evidence by lecture ID so identical objective IDs in two lectures do
not merge. Exclude invalid lectures and preserve authored order in queue links.

Refresh due state on mount, `visibilitychange`, window focus, and a bounded
interval while a queue is mounted. The pure due predicate remains injectable
and deterministic in tests.

Choice retry must be explicit. Thread both stable `assessmentId` and DOM
`anchor` through `RenderBlocks`, `Quiz`, `QuestionSet`, and
`AssessmentShell`. Add a `retry()` contract that clears only the mounted choice
lock; the next check replaces the existing answer attempt under the current
answer key. Retry alone does not create history or mutate the review schedule.
After checking, the learner rates the item; abandoning a retry keeps the prior
schedule.

Use the existing no-JavaScript fallback pattern. Without hydrated learner state,
render the authored page and a neutral state rather than an invented queue.

### 6. Update authoring and reviewer documentation

Update only authored examples and documentation, never raw-source evidence:

- `README.md` — document objective IDs, objective references, local review
  state, and the non-grading posture.
- `SKILL.md` and `.codex/skills/lecture-site-engine/SKILL.md` — teach agents to
  add explicit objective IDs and assessment references when source supports
  them, without fabricating learning objectives.
- `content/lecture.template.md` and non-raw example templates — add a small
  source-grounded example of objective IDs and `objective_refs`.
- `docs/mvp-review-checklist.md` — add objective-ID and assessment-coverage
  review checks if this remains the project launch checklist.

## Risks, Assumptions, And Mitigations

- **Objective syntax migration risk:** changing `string[]` to structured
  objectives can affect many consumers. Use a compatibility normalization layer,
  a list-item-specific locator model, update tests together, and preserve visible
  output for unmarked bullets.
- **False mastery risk:** a correct answer is not proof of durable learning.
  Use evidence language, avoid scores, and document status semantics.
- **Scheduler complexity risk:** overfitting a learning algorithm would expand
  scope. Keep the schedule pure, bounded, explainable, and user-controlled.
- **Storage drift risk:** authored IDs and assessment IDs may change. Validate
  state against the current normalized registry and discard stale entries.
- **Hydration risk:** due counts must not flash incorrectly before local state is
  loaded. Render a neutral loading/unknown state until the provider is ready.
- **Provider complexity risk:** do not turn `ProgressProvider` into an
  untestable god component. Prefer a composable review provider or pure hooks.
- **Privacy risk:** learner state is local-only and must not enter validation,
  static export, review packages, or logs beyond existing safe warnings.
- **Content-quality risk:** objective coverage diagnostics must be advisory
  unless an explicit reference is malformed or unresolved.
- **Raw-source risk:** examples and authored templates may change; raw lecture
  evidence must never be created, edited, summarized, or overwritten. Protected
  paths are `content/raw-lecture.txt`, `lectures/*/raw-lecture.txt`,
  `lectures/raw-course.txt`, and raw-source fixtures under `examples/`.
- **Current-content migration risk:** existing derived lecture templates use
  unmarked objectives. The launch metric applies to new or deliberately
  migrated fixtures; do not silently rewrite every existing lecture.
