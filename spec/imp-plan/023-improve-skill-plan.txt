# Implementation Plan: Improve the Research-Backed Lecture Authoring UX

## Summary

Improve the Lecture Site Engine agent skill for the interaction shown in Spec 023. The current flow correctly notices that the starter raw file is only a placeholder, but then has two poor user-experience transitions:

- it turns “create lecture for DB Internals” into a source-blocking request without explaining the available authoring paths;
- after the user says “I don't have source find it in internet,” it researches immediately and silently treats the existing 01-introduction scaffold as the scope, producing one introductory lecture without first resolving whether the user wants a standalone lecture or a broader course.

The revised skill should provide a short, staged authoring intake and progress flow. It should report the current source/evidence state, detect when a request is broad enough to represent a course, ask one targeted lecture-vs-course question when that choice materially changes the output, and recommend a multi-lecture collection for umbrella topics without creating one silently. After explicit internet-research authorization, it should produce a concise research brief and proposed outline/source basis before drafting when scope is ambiguous, then generate the selected single lecture or collection. It should finish with a clear report of files, lecture count, validation status, human-source limitation, and next action.

This remains an agent-guidance improvement, not a new interactive product workflow. The engine will not fetch URLs at runtime, persist authoring-session state, create raw evidence, or add a new citation schema. Existing collection scaffolding, course metadata, validation, source review, doctor, review-package, and resource_links capabilities should be reused.

## Proposed Implementation Steps

1. Define a source-aware authoring intake in the canonical skill.
   - Start every authoring request by reporting the active mode, relevant template/scaffold paths, raw-source status, and whether a real source is available. Treat scaffold placeholders as non-evidence.
   - Classify the request as a standalone lecture or a course/collection request based on the wording and requested scope. Treat umbrella topics such as “DB Internals” as potentially course-level rather than assuming the first scaffold lecture is the complete result.
   - When the choice materially affects output, ask one targeted question with a recommended option, for example: “Do you want one introductory lecture or a multi-lecture DB Internals course? I recommend a collection because the topic is broad.” Do not ask a long questionnaire.
   - If the request is sufficiently clear, state the assumption and proceed. Never let lectures/01-introduction silently decide course size, topic boundaries, audience, or lecture count.
   - Explain the two source paths: user-provided human raw source, or agent-time external research only after direct user authorization. Missing/placeholder source alone must not authorize browsing or source fabrication.

2. Define a staged external-research and authoring flow.
   - Stage 1 — Authoring brief: summarize the selected mode, working title, audience/level assumptions, source strategy, expected output paths, and the immediate next step.
   - Stage 2 — Research brief, only after explicit internet authorization: use authoritative primary/current sources, keep the research bounded to the requested scope, identify the proposed concepts/sections, and list concise source links. Do not write research notes or copied web content into raw-source files.
   - Stage 3 — Scope checkpoint: when the topic or lecture-vs-course choice remains material, show the proposed outline and source basis and wait for the user’s confirmation. When the request is clear and authorization is explicit, state the outline/assumptions and proceed without unnecessary extra turns.
   - Stage 4 — Draft: generate the selected standalone template or collection. For a collection, use lectures/course.yaml and numbered lecture templates in authored order; create only the number of lectures justified by the confirmed outline. For a standalone lecture, do not expand it into a course.
   - Stage 5 — Verify and hand off: run validation, source review, doctor, and review-package steps when requested. Report exactly what was created/updated, lecture count, validation result, source/evidence status, warnings, and the next command or user decision.
   - Keep progress messages user-facing and outcome-oriented. Do not expose confusing internal exploration, speculative scope changes, or a “created” success claim before validation and artifact paths are known.

3. Define the provenance and research-output contract.
   - Preserve the existing prohibition against creating, editing, rewriting, summarizing into, replacing, deleting, or overwriting content/raw-lecture.txt, lectures/<slug>/raw-lecture.txt, lectures/raw-course.txt, or raw-source fixtures.
   - Label an externally researched result as a derived draft, not human source evidence. A missing or placeholder raw file remains missing/placeholder; it is never promoted to present.
   - Use the existing resource_links component for concise references in the generated lecture/collection. Do not add a citation field, bibliography schema, URL crawler, link checker, or runtime fetch promise.
   - Tell the user that source-fidelity approval still requires human review/evidence. Preserve the existing limitation that the system cannot cryptographically determine whether supplied text was AI-generated.
   - Keep lectures/raw-course.txt as optional shared human evidence whose presence is not authorization to load it into agent context; it is separate from internet-research permission.

4. Align distributed guidance and user/maintainer documentation.
   - Update .claude/skills/lecture-site-engine/SKILL.md as the source of truth, then align .codex/skills/lecture-site-engine/SKILL.md, SKILL.md, AGENTS.md, and CLAUDE.md.
   - Update README.md with a compact authoring flow: inspect source status, choose standalone vs collection, authorize research if needed, review the brief/outline, generate, validate, and hand off. Explain that the starter 01-introduction folder is only a scaffold.
   - Update DEVELOPMENT.md with the agent-time-only boundary, no-session-state/no-runtime-fetch decision, reuse of existing collection and review commands, and unchanged source/readiness semantics.
   - Include an example of the preferred response shape for a broad internet-backed request, without hardcoding DB Internals content or adding external web text to fixtures.
   - Preserve the golden-conversion restriction: do not expose examples/golden.template.md in agent-facing authoring guidance.

5. Add deterministic UX-contract tests and verify compatibility.
   - Extend tests/lecture-template/agent-guidance.test.ts to assert that all five guidance entry points describe:
     - source-status inspection and placeholder non-evidence;
     - lecture-vs-course scope selection and the rule that the starter scaffold does not determine course size;
     - one targeted clarification/recommended option for materially ambiguous scope;
     - explicit authorization before internet research;
     - research brief/outline/source-link and final artifact/status reporting;
     - derived-draft labeling, raw-source protection, shared-source opt-in, and provenance limitation.
   - Keep tests deterministic and text-contract-focused. Do not run network requests, simulate an LLM, test search quality, or attempt to prove whether content is AI-generated.
   - Preserve init.test.ts and scaffold.test.ts coverage proving placeholders are created/classified correctly, lectures/raw-course.txt is not auto-created, and authored files are not overwritten.
   - Preserve source-review, doctor, and review-package tests proving missing/placeholder human evidence remains reportable and nonfatal where currently specified, and external resource links remain authored metadata.
   - Run the full validation, lint, typecheck, test, application-build, and CLI-build checks. Use a disposable manual interaction to verify both a clear standalone request and a broad “find it online” request: the former does not expand into a course; the latter presents the recommended scope/outline before generating a confirmed collection or lecture set.

## Data Model, API, Persistence, and Event Changes

- No lecture-template frontmatter, component, course metadata, route, or collection schema changes.
- No new agent-session state, interactive CLI, browser UI, persistence, database, event, analytics, or remote research service.
- Reuse existing lectures/course.yaml, numbered lecture directories, validation, source-review, doctor, review-package, and resource_links contracts.
- No changes to present, missing, or placeholder raw-source statuses or source-fidelity readiness semantics.
- Internet access remains an explicitly authorized authoring-agent action only. The generated site renders authored links without fetching them.

## Risks, Assumptions, and Open Questions

### Risks and mitigations

- The new flow could become too conversational. Limit it to one targeted scope question and one outline checkpoint only when the choice materially changes output; otherwise state assumptions and proceed.
- Broad topics may still be under-scoped. Recommend a collection for umbrella topics but do not create multiple lectures without a confirmed outline or an explicit user-approved default.
- The starter scaffold may continue to bias the agent toward one lecture. State in every relevant guidance entry point that 01-introduction is a placeholder, not a course plan or lecture-count decision.
- External research may be mistaken for source fidelity. Label the result as derived, preserve raw-source statuses, and report human-source limitations in the final handoff.
- Progress messages may expose internal tool failures or false success. Require outcome-oriented stage reporting and only claim completion after validation/artifact checks.
- Guidance entry points may drift. Keep the canonical Claude skill authoritative and extend the existing deterministic alignment test.
- Manual UX verification may depend on a particular agent surface. Keep the product contract in portable guidance language and test observable repository behavior separately from the conversational transcript.

### Assumptions

- The user’s “I don't have source find it in internet” is explicit authorization for agent-time research, but not permission to choose an unconfirmed course scope.
- “DB Internals” is representative of a broad umbrella topic where recommending a multi-lecture collection is better UX than silently creating only 01-introduction.
- Existing scaffolding and collection commands are sufficient for generation once the agent has a confirmed outline; this spec does not require new CLI prompts or a course-planning data model.
- External research can be represented through existing lecture content and resource_links; no raw-source or bibliography ingestion feature is required.
- Existing raw-source ownership and review semantics from Spec 022 remain authoritative.

### Open questions

- No blocking implementation question remains for this plan. The recommended default is one concise lecture-vs-course clarification for broad topics, with a collection recommendation; implementers should not auto-generate a multi-lecture course when the user has not confirmed the scope.

## Out of Scope

- Implementing DB Internals lecture content or selecting its final curriculum in this repository.
- Automatically browsing the internet because a raw file is missing.
- Creating, editing, summarizing into, replacing, deleting, or overwriting raw-source files.
- Treating external web content or generated research notes as human present source evidence.
- Adding an interactive course planner, conversational state machine, new CLI prompts, persistent authoring sessions, citation/bibliography schema, URL crawler, link checker, runtime fetch, hosted research service, or AI-authorship detector.
- Changing lecture components, validation rules, collection routes, learner progress, static export, review-package paths, or doctor readiness semantics.
